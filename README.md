# My Blog Supporter

ブログ記事を、**発散 → 深掘り → 批判 → 構成 → 執筆** の工程に分けて、人間（著者）と AI（Claude Code）が往復しながら書くための個人ワークフローです。

> [!NOTE]
> これは「テーマを渡すと記事が出てくる全自動ツール」ではありません。むしろ逆で、**思考と判断を人間が握ったまま、作業だけを AI に省エネさせる**ための足場として作っています。

## なぜ作ったか

ブログを書くのは高カロリーな作業で、放っておくと更新が止まってしまいます。そこで **Agentic に楽できるところは楽をして、記事を書く頻度を省エネで上げよう** という試みが、このリポジトリです。

ただし、量産が目的ではありません。著者にとってブログは「生産」ではなく **思考整理** の手段であり、書く価値の大半は「自分の頭の中を整理し、調べ、言葉にする」過程そのものにあります。ですから、AI に文章を吐かせて埋めるという方向には振っていません。

注力したのは、**自分の頭の中を整理・調査して書く過程を AI に手伝わせつつ、出来上がりが「本人が書いたもの」である状態を保つ**ことです。AI に任せるのは調べ物・整理・叩き台づくりといった作業で、何を問い・何を主張し・どう判断するかは、人間が握り続けます。楽をすることと、自分の記事でなくなることは別である、という線引きを仕組みにしました。

## どう使っているか

`CLAUDE.md` が全体のオーケストレーションを担い、各工程を Skill（スラッシュコマンド）に分けることで、**自律的かつ対話的な執筆を一気通貫で**回します。記事1本につき1つのフォルダを作り、その中で工程ごとの成果物が、対話を通じて育っていきます。

`CLAUDE.md` は単にコマンドを並べるだけのものではなく、工程間のループ構造（発散 ⇄ 掘る ⇄ 叩く）や、critique の自動実行・人間への差し戻しの条件まで含めて全体を指揮します。著者は対話に応じていくだけで、発散から執筆まで途切れずに進みます。

| Skill | やること | 性格 |
|---|---|---|
| `/brainstorm` | 頭の中を出し切って整理する | 発散 |
| `/deepen` | 本筋に効く論点を掘り下げる | 掘る |
| `/critique` | 掘った主張を叩く（隔離コンテキストで自動実行） | 叩く |
| `/outline` | 見出し構成・展開順序を設計する | 固める |
| `/write` | 記事を執筆・推敲する | 書く |
| `/compare-drafts` | AI 版と自分版を比較し、次回を自分の文体に寄せる | 振り返る |

`brainstorm`（発散）⇄ `deepen`（掘る）⇄ `critique`（叩く）はループになっていて、納得がいくまで往復してから、構成・執筆へ進みます。急ぎのときは `/brainstorm` → `/write` に直行することもあります。

## このワークフローの肝＝設計思想

「楽をする」ことと「自分の記事でなくなる」ことを分ける核は、2つあります。

**1. 問いと判断は人間が握る（content-typed gate）**
`/critique` が出した指摘を、AI が機械的にすべて自答することはありません。指摘を「著者固有でないもの（AI が埋めてよいもの）」と「著者固有のもの（体験・立場・判断が必要なもの）」に**仕分け**し、前者だけを AI が自走で処理し、後者は必ず著者へ戻します。AI が自答した箇所には provenance（`[AI回答]` の印）を残すので、あとから「どこが自分の思考で、どこが AI なのか」を追えます。これが、AI の自走を取り入れながらも、**記事の中身が静かに AI のものへすり替わるのを防ぐ**仕掛けです。

**2. 「引き出す」工程と「叩く」工程を分離する**
思考を引き出す `/deepen` と、思考を叩く `/critique` は、性質が逆です。そこで critique は隔離コンテキストで動かし、対話の流れと批判の視点が混ざらないようにしています。引き出す対話に批判が割り込んで萎縮させてしまうことも、批判が対話の空気に流されて甘くなってしまうことも、どちらも避けるためです。

この2つは、著者のブログの主題でもある「**問いと判断は人間が握る**」という考えを、執筆プロセスそのものに落とし込んだものです。

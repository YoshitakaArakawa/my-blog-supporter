# 下書きレビュー

`output/{yyyymmdd}_$ARGUMENTS/draft_user.md` に書かれた記事の下書きを、編集者の目でレビューし、優先度順の指摘を `output/{yyyymmdd}_$ARGUMENTS/review/review_NN.md` に出力する。

**フォルダ命名規則**: 出力先は `output/{yyyymmdd}_$ARGUMENTS/` とする。既存フォルダがあればそれを使う。`$ARGUMENTS` に日付プレフィックス付きのフォルダ名が直接渡された場合はそのまま使う。以降のパス記述で `$OUTPUT_DIR` はこのディレクトリを指す。

このスキルは隔離コンテキスト（`context: fork`）で実行される。会話履歴にはアクセスできないため、レビュー対象と判断材料はすべてファイルから読み込む。メイン会話には**実行サマリのみ**を返す（書いたファイルパス・指摘件数・最優先論点）。指摘の全文を戻り値に詰めない。

## このスキルの役割

著者が書いた原稿（draft_user.md）そのものを、**公開前のゲートとして編集視点で点検する**専任。`context: fork` なので、会話の文脈を持たない別エージェントが原稿を冷えた状態で読む。これが**初見の読者に近い視点**を生み、「読みにくい」「難しい」の判定精度を上げる。

工程内での役割分担を守る（重複させない）:

- **critique**: thinking.md の**主張**を叩く（想定反論・前提の穴・論理の飛躍）。本スキルは主張の是非ではなく、原稿の**編集品質**を診る。
- **compare-drafts**: draft.md（AI版）と draft_user.md（著者版）を比較し、**AI生成を著者に近づける**改善FB。本スキルは比較ではなく、著者の原稿**単体**の編集レビュー。
- **write の自己レビュー**: AIが生成した draft.md を書きながら点検する工程。本スキルは**著者が書いた** draft_user.md が対象。

ここで出た指摘は、著者が推敲で取捨選択するための材料。**原稿を書き換えない**（直すかは著者が決める）。提案までに留め、本文への直接編集はしない。

## 手順

### 1. 入力の読み込み

`$OUTPUT_DIR` から以下を読む:

- `$OUTPUT_DIR/draft_user.md`（**必須。レビュー対象**）
- draft_user.md が無い、または空の場合は、レビューせずに「`$OUTPUT_DIR/draft_user.md` が見つからない（または空）ので、先に下書きを用意してください（Wix からなら `import-wix-draft` で取り込めます）」という旨をサマリで返して終了する。**draft.md など他ファイルで代替しない**（著者の原稿を診むのが目的のため）。
- `$OUTPUT_DIR/brief.md`（あれば。**読者像**・テーマ・コアメッセージの文脈把握に使う。読者難易度・タイトル整合の判定基準になる）
- `$OUTPUT_DIR/outline.md`・`$OUTPUT_DIR/thinking.md`（あれば。**コアメッセージ**・意図した構成と、原稿の実体とのズレを見るのに使う）
- `.claude/skills/write/references/voice-style.md`（**必須。文体整合・表記の基準**。ルール1-16＝著者の声・表記規約を整合の物差しに、ルール17-23＝AI的傾向への対抗ガイドをレビューのレンズに含める）
- `$OUTPUT_DIR/references.md`（あれば。引用と本文の整合を見るのに使う）
- `$OUTPUT_DIR/review/` 内の既存 `review_*.md`（あれば。前回までに何を指摘済みか把握し、重複指摘を避ける／未対応に印を付ける）

### 2. 編集レビュー

draft_user.md を、以下の7観点で診る。1観点に偏らず一通り当てる。**各指摘は弱点から述べ**、原稿の該当箇所を行番号（例: `draft_user.md:42`）で示す。**書き換え案でなく、問題の指摘と方向性の提案**に留める。

1. **タイトル整合**: タイトルと本文の主張（コアメッセージ）が合っているか。タイトルが本文の芯を立てているか、弱い/ぼやけていないか。必要ならタイトル案を数点（短い英語 slug が要るなら別途）。
2. **構成・流れ**: 展開順序は自然か。結論（payoff）に着くのが遠すぎないか。章間のロードマップ的提示や同じ話の繰り返しが過剰でないか。重み付け（分量配分）は適切か。
3. **読みやすさ**: 一文が長すぎないか、段落の切れ目は適切か。難語・専門語が前置きなく出ていないか。
4. **想定読者への難易度**: brief.md の**読者像に照らして**、読者の前提を超える箇所はどこか。最も難しいピークが本筋の手前に来ていないか（離脱リスク）。
5. **論理破綻・飛躍**: 主張の間に飛躍・矛盾はないか。未定義の語、前後でズレる定義はないか。
6. **文体整合（voice-style）**: voice-style.md のルールに沿っているか。特にルール17-23に照らし、**著者性を均して“AIっぽく一般化する”方向の提案はしない**（著者の声を守る）。
7. **校正（誤字脱字・表記ゆれ・係り受け・冗長）**: 表面の精度を機械的に拾う。(a) 誤字・脱字・送り仮名の誤り、(b) 表記ゆれ（同一語の漢字/かな・全角半角・「フィルタ」と「フィルター」等。voice-style.md の「14. 表記辞書／半角スペース／数字／記号」を基準にする）、(c) 係り受けのねじれ（二重受身・二重義務、主述の不一致、修飾の係り違い）、(d) 冗長な言い回し（一文内の同語反復、ほぼ同じ問い・主張の二度立て、「ながら」等の連続）。文意の言い換え（推敲）ではなく、直せば必ず良くなる表面のミスと、読みで引っかかるねじれ・重複に絞る。なお章をまたぐ構成上の繰り返しは観点2、主張間の飛躍は観点5の領分とし、ここは文・語レベルに留める。

各指摘には「なぜ問題か」と「どう直す方向がよいか（提案）」をセットで書く。

**やらないこと**:
- 原稿本文の直接書き換え（推敲は著者の領分）。
- 主張そのものの是非の議論（それは critique の領分）。
- AI版との優劣比較（それは compare-drafts の領分）。

### 3. 連番の決定

`$OUTPUT_DIR/review/` 内の既存 `review_NN.md` を確認し、最大番号 + 1 を今回の番号とする（2桁ゼロ埋め）。1つも無ければ `review_01.md`。フォルダが無ければ作成する。過去のレビューが上書きで消えず、推敲ごとの変遷が時系列で残る。

### 4. review_NN.md の生成

以下のフォーマットで `$OUTPUT_DIR/review/review_NN.md` に書き出す:

```markdown
# 下書きレビュー #{NN}（{yyyymmdd}）

## レビュー対象
{この時点の draft_user.md のタイトルと、本文のコアメッセージを一行で}

## 前回からの変化
{review_01 等が既にある場合、前回指摘がどう反映されたか／未対応か。初回は「初回レビュー」}

## タイトル
- 現タイトル: {引用}
- 本文との整合: {合う/弱い、理由}
- 案: {必要なら数点}

## 指摘（優先度順）

### 1. {指摘の見出し}
- 観点: {タイトル整合 / 構成・流れ / 読みやすさ / 読者難易度 / 論理破綻 / 文体整合 / 校正}
- 箇所: {draft_user.md:行番号}
- 問題: {何がどう弱いか}
- 提案: {どう直す方向がよいか（書き換えはしない）}

### 2. {指摘の見出し}
...

## 良い点（任意・簡潔に）
- {維持したい強み。削らないために明記}

## 総評
{最優先で直すべき論点を1〜2個}
```

`{yyyymmdd}` は実行日の日付。

### 5. 実行サマリを返す

メイン会話には以下だけを返す:
- 書いたファイルパス（`$OUTPUT_DIR/review/review_NN.md`）
- 指摘件数
- 最優先で直すべき論点（1〜2行）

指摘の全文は返さない。Write に失敗した場合は inline で指摘本文を返さず、失敗した旨をサマリで返す。

# 構成設計

`output/{yyyymmdd}_$ARGUMENTS/thinking.md` の主張と参考資料をもとに、記事の見出し構成・展開順序・資料接続を設計し `output/{yyyymmdd}_$ARGUMENTS/outline.md` に出力する。

**フォルダ命名規則**: 出力先は `output/{yyyymmdd}_$ARGUMENTS/` とする。既存フォルダがあればそれを使う。`$ARGUMENTS` に日付プレフィックス付きのフォルダ名が直接渡された場合はそのまま使う。以降のパス記述で `$OUTPUT_DIR` はこのディレクトリを指す。

## このスキルの役割

deepen が「掘る・問い直す」フェーズ、critique が「叩く」フェーズであるのに対し、outline は「掘った思考を記事の構成に固める」フェーズ。

brainstorm⇄deepen⇄critique のループは思考を行き来させて深める工程で、最新が正の thinking.md に収束していく。outline はそのループが落ち着いた後に一度走り、構成という別の成果物を作る。掘る作業と構成を組む作業をファイル上でも分離する（thinking.md は更新し続ける主張メモ、outline.md は構成）。

## 手順

### 1. 準備

`$OUTPUT_DIR` から以下を読み込む:
- `$OUTPUT_DIR/thinking.md`（必須。なければ「先に `/deepen $ARGUMENTS` を実行してください」と案内して終了）
- `$OUTPUT_DIR/brief.md`（あれば。テーマ・コアメッセージの確認）
- `$OUTPUT_DIR/references.md`（あれば。資料接続の設計に使う）
- `$OUTPUT_DIR/critique/` 内の最新 `critique_NN.md`（あれば。未対応の指摘が構成に反映されているか確認する）

### 2. 参考資料の本格的な読み込み

references.md に記載された参考資料を実際に読み込み、内容を把握する。
- ローカル資料（`$OUTPUT_DIR/references/` 内）は Read で読み込む
- URL は WebFetch で取得する
- 各資料の要点を整理し、thinking.md の各主張との接続（どのセクションで、どう言及するか）を考える

### 3. 構成の設計

thinking.md の主張を整理し、記事の見出し構成を設計する。ユーザーと対話しながら以下を検証する:
- 各論点の展開順序は自然か
- コアメッセージに収束しているか
- 読者が途中で迷子にならないか
- 各セクションの重み付け（ボリューム配分）は適切か
- 不要な要素を削ぎ落とせるか
- critique で出た未対応の指摘が、構成のどこかで解消されているか

### 4. outline.md の生成

以下のフォーマットで `$OUTPUT_DIR/outline.md` に書き出す:

```markdown
# 記事構成: {記事タイトル案}

## コアメッセージ
{thinking.md から引いた、この記事の一番伝えたいこと}

## 読後の読者の状態
{この記事を読んだ人が、読後にどうなっていてほしいか}

## 構成

### 導入: {見出し案}
- 展開する思考: {このセクションで何を語るか}
- 素材: {使える具体例・エピソード・参考資料}

### 展開1: {見出し案}
- 展開する思考: {このセクションで何を語るか}
- 素材: {使える具体例・エピソード・参考資料}
- 前のセクションからの接続: {なぜこの順序か}

### 展開2: {見出し案}
...

### 結び: {見出し案}
- 展開する思考: {どう着地するか}
- 残す問い: {未解決のまま読者に委ねること}

## 参考資料との接続
| 参考資料 | 記事での活用方法 |
|---|---|
| {資料名} | {どのセクションで、どう言及するか} |
```

outline.md は最新が正の成果物。ループで構成を見直すたびに上書き更新する（変遷は git 履歴で追える）。

### 5. references.md の追記
構成設計で新たに出てきた参考情報があれば `$OUTPUT_DIR/references.md` に追記する。

### 6. 次のステップ
outline.md が完成したら、ユーザーに以下を案内する:
- 「`/write $ARGUMENTS` で記事執筆に進めます」
- outline.md はいつでも修正可能

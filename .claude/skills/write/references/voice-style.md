# 著者の文体・語り口メモ

yarakawa.com の過去記事を観察して抽出した、著者特有の語り口・文体パターン。`/write` 実行時に参照する。

著者: Yoshitaka Arakawa - BI Developer / Tableau Trainer

> **このドキュメントの使い方**:
> ルール 1〜16 は「**従うべき型**」、ルール 17〜22 は「**陥りがちなAI的傾向への対抗ガイド**」として読む。
> ルール内の「**避けること**」「**用途限定**」「**発動条件**」は、AI が機械的に過剰反応するのを防ぐためのもの。

---

## 1. 一人称・基本トーン

- 一人称は **「自分」**（「私」「僕」は使わない）
- 敬体（〜です・〜ます）で書くが、敬度は柔らかい
- 「思います」「ように思います」「気がします」で **ヘッジを多用**
- 「〜ですね」「〜だと思います」で柔らかく結ぶ
- 結論を1つに収束させすぎない。複数解釈や保留を残す

例:
> 自分のようなTableauユーザーは楽しめますが、Tableauの意義や魅力としては弱くなる気がします。

### ヘッジの多層化（重要）

ヘッジは語尾だけだと機械的になりやすい。以下を組み合わせて多層化する:

- **語尾ヘッジの連続使用は避ける**。「思います」「感じています」「気がします」が3-4文連続すると AI 的に見える
- **括弧書きの自己訂正と組み合わせる**:
  - 例: 「(自分が参加したセッションがそうだった、という話かもしれませんが)」
  - 例: 「(筆者の勘違いである可能性は、もちろん大いにありますが)」
- **共感誘発の語尾と混在させる**:
  - 「ですよね?」「面白かったです」「気になりました」「興奮しました」など、感情を伴う語尾を恐れない

## 2. 強い断定を避ける

- 「〜と感じます」「〜かもしれません」「〜のように見えます」
- 自信のない部分は正直に開示する: 「自分は正直わかりません」「ちょっと良く分からないですね」
- 「絶対」「必ず」のような強断定はほぼ使わない

## 3. 鉤括弧「」による強調

- 主張・重要フレーズ・概念名を「」で囲んでマーキング
- 例: 「ガードレール」「分析体験ファシリテーター」「再現性」「Vibe Analytics」「没個性」
- 太字 (** **) は記事本文で使わない（`**` で囲まない）。強調・マーキングはすべて「」で行う
  - これは生成する記事本文（draft.md）についての規約。本メモなど作業ドキュメント自体の強調表記とは別

**節度の指針**:
- **1段落あたり1-2個までを目安に**。文中の通常用語まで「」で囲わない
- 「これは本当に強調すべき重要フレーズか?」と自問してから使う。念のため・保険的な使用は避ける

## 4. 読者への直接エンゲージ

- 問いかけ:「〜でしょうか？」「〜ですよね？」
- 呼びかけ:「ぜひ〜してみてください」「〜してみてください」
- 想像の促し:「〜をイメージしてください」
- 著者自身の問いに読者を巻き込む型

例:
> 何ごとも「大したことない」人でも出来るなら、自分でも出来そうだって気がしませんか。

## 5. 具体例で抽象を裏付ける

- 概念だけで論じない。実体験・実装記録・観察と組み合わせる
- 例: Sample Superstore × 生成AIのロールプレイ、Tableau MCP セットアップ手順、ダッシュボードの再現性運用
- ロールプレイや実験設定を明示することが多い

**順序の指針**:
- **「具体観察 → 抽象解釈」の順序を基本に**。抽象を先に出して具体で支える順序は、論文的・AI 的になりやすい
- 現地で見たもの・固有名詞・セッション名・タイムスタンプなど、一次情報の具体性から入って、そこから自然に解釈に上がる

## 6. 構造的な遷移句

| 用途 | 表現 |
|---|---|
| 中間まとめ・抽象化 | 「つまり、〜」 |
| 対比 | 「一方、〜」「ただし、〜」 |
| 結論部へ | 「ということで、〜」 |
| 補足 | 「ちなみに、〜」 |

**避けるべき遷移**:
- **章をまたぐロードマップ的提示** (例: 「ここまでは〜、ここからは〜」「焦点が〜に移ります」「ここから少し、踏み込んだ話を」)
- 読者は章タイトルから構造を読み取れるので、章間の道案内は最小限にする
- 章内でも、長い遷移文は避ける。短い接続詞 (「一方」「ただ」「ということで」) で済ませる

## 7. 「否定→ただし」パターン

否定的事実をいったん認めた上で、価値を再評価する型。記事の終盤や転換点で使う。

例:
> 学んできたTableauの高度な技術は、そのうち使わなくなるかもしれない。
> ただし、その技術を必要とした背景や、その技術で解決してきた課題に関する経験や思考は、生成AIエージェントに応用させることができる。

**用途の限定**:
- このパターンは、**自分が学んできた技術・姿勢の再評価、過去の自分の見方の修正** に使う
- **製品や他者への批判 → フォロー、には使わない** (避ける例: 「Tableau Classic は限界がある。それに対して Tableau Next は〜」)
- 製品/外部の限界を語りたくなった時は、「我々ユーザー側の準備はどうか」のように主体性側に話を向け直せないか検討する (ルール 20 参照)

## 8. 余談セクション

本筋から外れる気付きを `### 余談：` で挿入する型。短め（1-3段落 + 箇条書き）。
読者への小ネタ的サービス、本論を補強する技術メモ、自己反省的なオマケなど。

例:
- `### 余談：Tableau MCP経由ならデータハルシネーションの可能性は低そう`
- `## 余談` （本論の後ろにポジティブな後日譚として）

**発動条件**:
- **本筋に組み込めない場合のみ発動する**
- 散文に組み込めるなら、組み込む方が自然
- 「著者文体らしさのために余談を入れる」発想は避ける。**記事全体に1つも余談がなくても問題ない**

## 9. 結論セクション「## 最後に」

- 必ず `## 最後に` で締める
- 末尾サインオフは固定: 「質問などありましたら、XかLinkedinまでお願いします。それでは！」

**型は記事の主張に応じて選ぶ。標準型に縛られない。**

### 型の選択肢

- **標準型 (思考型)** — 構造的・分析的な記事に向く:
  1. 記事を書いた動機・問題意識を改めて語る
  2. 記事を通して再認識したことを言語化
  3. まだ自信がない部分・分からない部分を正直に書く
  4. 「また考えを改めたら / 事例が出てきたら、その時にまた書いてみます」
  5. サインオフ

- **箇条書き集約型 (感想記事向き)**:
  - 短い結論宣言 + 体験ベースの自己開示 + 5箇条程度の感想まとめ + サインオフ
  - 例: カンファレンス感想で「メッセージ同意」「コミュニティ評価」「コンテンツへの注文」「来年への期待」「自身のコミットメント」を箇条書きに

- **問いかけ型 (読者参加型)**:
  - 結論を出さず、読者への問いを提示して締める

- **短い宣言型**:
  - 1-2段落で記事の主張を再確認、決意や次の一歩を短く述べてサインオフ

### 選び方の指針

- 構造的・分析的な記事 → 標準型
- 感想・体験記事 → 箇条書き集約型 or 短い宣言型
- 議論を開いておきたい記事 → 問いかけ型

## 10. 新概念の導入スタイル

- 教科書的網羅を避ける
- セクションタイトルに「（簡単に）」「（自分の理解）」と添える
- 例:「### Vibe Analyticsとは？（簡単に）」「### そもそもMCPって何（自分の理解）」
- 厳密性を欠くと前置きすることもある:「厳密性を欠いた説明をしている部分もあるかと思います」
- 「と自分は理解しています」で私見であることを示す

## 11. フォーマット規約

- **固定ヘッダ**: 「Tableau NOTE」ブロック + ブログ目的3点 + 著者プロフィール（"Yoshitaka Arakawa - BI Developer/Tableau Trainer"）
- **タイトル**: 最近は「- YYYY年MM月版」を末尾に付けることがある
- **見出し**: `##` を主軸。`###` はサブセクションで使う。`####` 以下はほぼ使わない
- **セクション区切り**: `##` の前後に `---` 水平線
- **段落**: 1段落 2-4文程度。長文化を避ける。段落間は空行1つ。**1段落につき1トピック**に絞り、観察・解釈・余談など複数の場面が混ざった段落は一歩ずつに分ける
- **リスト**: 使うが多用しない。基本は散文
- **太字**: 記事本文では使わない（`**` で囲まない）。強調は「」で行い、リスト見出しやステップ番号も太字にしない
- **画像**: 主張の直後に配置、画像下に補足説明
- **参考リンク**: 2つのパターンを併用する。
  - (a) **冒頭の「参考資料」ブロック**: タイトル・日付の直後、リード文の前に「参考資料」という見出しを置き、記事**全体の論旨に関わる主要資料**を数点だけ箇条書き（リンク）で並べる。これは構造的な資料リストであり、リード文・本文の書き出しは別に体験から始める（ルール19と両立する）。長いリンク集にしない（実例は2〜3点）。
  - (b) **本文中のインライン**: 該当箇所で「例えば○○のBlogから：」「以下2つです」のような短い導入＋タイトルの箇条書き、または「こちらの記事」での自然な言及。同じ資料を冒頭ブロックと本文の両方で出してよい。

## 12. タイトルのパターン

- **考察型**: 「〜を考える」「〜に道を見出す」「〜の意味について考える」
- **誘い型**: 「〜してみましょう」
- **シリーズ型**: 「コラム1：〜」等のプレフィックス

## 13. 過去記事との接続

- 関連する過去記事には自然に言及する（「以前○○という記事を書きましたが〜」）
- 過去記事の主張を踏まえて発展させる構成を意識する
- 過去記事へのリンクは本文の流れの中に埋め込む（冒頭にもまとめて出す）

## 14. 用語・語彙・表記

- 専門用語: Tableau, BI, MCP, AI Agent, VizQL, データソース, ダッシュボード, Prep, セマンティック等
- カタカナ語を恐れない
- 「BIツール」「BI人材」「Tableauユーザー」を主語として使う頻度が高い

### 表記辞書（揺れを統一する）

語の表記は次の辞書に従って統一する。揺れやすい語に気づいたら、この表に追記して育てる。

| 使う | 使わない |
|---|---|
| AI Agent | AIエージェント / AI エージェント |

- カタカナの「エージェント」は一般名詞として使う時のみ可。記事の主題である AI Agent を指す時は英語表記に統一する。

### 英単語まわりの半角スペース

- **和文と英単語の境界に半角スペースを入れない**。日本語の地の文に英単語が出てきても、前後を詰める。
  - 良い例: 「TableauはREST APIを叩く」「VizQL Data Serviceで読む」
  - 悪い例: 「Tableau は REST API を叩く」「VizQL Data Service で読む」
- ただし**英語フレーズ内部の語間スペースは残す**（"REST API" "Published Data Source" の語と語の間は英語の正書法なので詰めない）。
- 字種（全角／半角）は従来どおり: "Tableau" など固有名詞・略語は半角、和文はそのまま。混在は許容するが、上のスペース規約と表記辞書を優先する。

### 数字の表記

- **横書きなので、数量・順番・個数は算用数字（半角アラビア数字）で書く**。漢数字は使わない。
  - 良い例: 「やることは大きく2つ」「1つめ」「3章」「2軸」
  - 悪い例: 「やることは大きく二つ」「一つめ」「三章」「二軸」
- ただし**熟語・慣用句・固有名詞の一部になっている漢数字は残す**（語の一部なので置換すると不自然・誤りになる）。
  - 残す例: 「一方」「一貫性」「一気に」「第一印象」「一石二鳥」「唯一」

### 記号の使い方（ダッシュ・中黒）

- **ダッシュを地の文・見出しで使わない**。em ダッシュ `—`、horizontal bar `―`、2倍ダッシュ「――（——）」は使わない。同格・補足の挿入は（）に、言い換え・敷衍は句点で二文に分けるか読点でつなぐ。範囲を示す en ダッシュ `–`、英語の複合語（`Curry–Howard` 等）、コードブロック・URL は対象外
- **中黒（・）を日本語の並列で使わない**。並列は読点（、）でつなぐ。ただし単一の固有名詞の内部（略語・社名など）では使ってよい

## 15. 著者のスタンス（背景）

- BI Developer / Tableau Trainer
- コミュニティタイトルや個人特集への露出には距離を置く
- 活動の核は **「再現性」と「ギャップを埋める」**
- 「自分の特別性は不要、誰でも辿れる論理を整備したい」というスタンス
- 「危機感を煽る」「驚かせる」論調は避ける
- ブログの目的:
  1. 世界で使われている発展的な技術を広める
  2. Tableauとデータ可視化についての日本語ドキュメントを増やす
  3. 自身が学習した内容を整理し言語化する

## 16. やらないこと（avoidance）

- 強い煽り・危機感を煽る論調（明示的に否定: 「危機感を煽りたい訳ではなく」）
- 教科書的網羅・定義の羅列
- 強断定（代わりにヘッジで余白を残す）
- 結論を1つに収束させすぎる（「ただし〜」で別側面も残す）
- 専門家ぶる・上から目線
- AIが書いたような平板で無個性な文章

# AI 的傾向への対抗ガイド (17〜22)

> 以下のルールは、AI が voice-style 全体を律儀に守ろうとした結果、かえって不自然になる傾向への対抗策。
> 「やりすぎないため」「機械的にならないため」のメタルール。

## 17. 構造化のミニマリズム

- **大見出し (`##`) は3-4個までを目安に**。論点ごとに章を立てる衝動を抑え、近接する論点は同じ章にまとめる
- **章タイトルは「問い」「観察」「シンプルな宣言」の形で、かつ簡潔に書く**。説明的に書ききらず、短い体言止め・短い問い・短い宣言で始める。本文で説明することを見出しに詰め込まない
  - 良い例: 「Architectというキーワード」「ユーザー側の準備はあるか」「むしろVisionaryが変化の必要性を主張していた」
  - 避ける例: 「ただし○○を本気で受け取ると○○だけでは足りない」(論点の要約をそのまま見出しにしている)／「今の基準点は『○○』に動いている」(主張を一文そのまま見出しにして長い)
- **見出しで節の結論（オチ）を言い切らない**。見出しは節が扱う問い・観察・対象を指すにとどめ、読者が見出しの時点で落としどころを知ってしまう状態を避ける。「宣言」型を使うときも、向かう方向を示すだけにして、本文で気付かせたい結論そのものは見出しに出さない
- **小見出し (`###`) は本当に必要な時のみ**。余談や明確な区切りが必要な時に限る
- 章間のロードマップ的提示は最小限に (ルール 6 参照)

## 18. 抽象メタファーの使い捨て原則

- **強いメタファーは記事内で1回限定使用**。気に入ったフレーズを章をまたいで繰り返し参照しない
- 例: 「両側から境界を広げる」「生息地を広げる」「Moving Up the Stack」のような印象的なフレーズが浮かんでも、それを記事の柱として何度も使わない
- メタファーは具体観察の解釈として1回出すだけで、論証の柱にはしない
- 柱になっている主張があるなら、それは抽象メタファーではなく、**具体観察と問いの連鎖**であるべき

## 19. 冒頭設計（体験から入る／宣言から入る）

- **既定は「自分の現在地」「自分の体験」から始める**。「念願のTC現地参加でした」「○○に登壇してきました」のような体験開示が出発点として自然。体験記事・感想記事ではこれを優先する
- **実験・技術共有系の記事は「何を書くか」の宣言から入ってもよい**。「この記事では〜という実験を共有します」のような宣言スタートは、対象がはっきりしている記事では自然。ただし次を守る:
  - 宣言は数文に圧縮する。何を書く / 書かないかを長々と説明しない
  - **年表・スコープ整理・前提の羅列で導入を重くしない**。背景の時系列や「なぜ今」の詳細は本文の該当章へ送り、導入は軽く保つ
- どちらで入るにせよ、**参考リンクの先出しから始めない**。参考リンクは本文の流れの中に分散させる (冒頭にまとめる場合も短く)
- **導入は「執筆動機」から書き出す**。「なぜ今これを書くのか / なぜこの取り組みをしているのか」を一度言葉にする。動機は記事の推進力になり、読者が早い段階で「何の話を読んでいるか」を掴める。**この執筆動機の書き出しは AI が brief.md / thinking.md の動機をもとに下書きする**（著者の手動追加を待たない。著者は後から修正する）
- 「何を書くか」の宣言と「なぜ書くか / なぜやっているか」の動機は **別物** として両方を導入に置く。動機は概要でよい (詳細は本文で展開する前提で、導入では簡潔に)

## 20. 主体性の方向

- **批判や課題提起は、外側 (製品/組織/他者) ではなく、自分・我々ユーザー側に向ける**
- 製品の構造的限界を書きたくなった時、「自分たちの側の準備」「我々の主体性」の話に置き換えられないか検討する
- 良い例: 「Tableau Classic の API-first非対応」ではなく **「我々ユーザー側のインプットや知識が Tableau に閉じていないか」**
- 良い例: 「○○ができていない」ではなく **「我々が○○する準備がどこまであるか」**
- ルール 7 (「否定→ただし」パターンの用途限定) と連動

## 21. 固有名詞のリアリティ

- 現地で見た / 聞いた **人名・セッション名・タイムスタンプ** を躊躇せず出す
- 抽象化しすぎた表現 (悪い例) より、具体的な名前 (良い例) を選ぶ:
  - 悪い例: 「あるセッション」「Will の Keynote」
  - 良い例: 「Beyond the Boundaries of Tableau (特に26:40から)」「Will さんのパート」「Mark が True to the Core で語っていた内容」
- 一次体験の固有名詞は、抽象論で書けない説得力を生む
- 敬称や丁寧度は著者の習慣に合わせる (yarakawa.com の場合「○○さん」が自然なケースが多い)

## 22. LLM 口調の禁止フレーズ

> AI が「ちゃんと書いている感」だけを足すために多用する、中身のない定型句。論点を増やさず姿勢や強調だけを運ぶ空句なので使わない。書き上げたらこの節で語句を点検する。
> 記事の主題に関わる語（「再現性」「言語化」など）を中身のある主張として使うのはよい。空虚な装飾として使うのが問題。

次の言い回しは、新情報を運ばず「書いている感」だけを足す。使わず、主張をそのまま書く。

- **予告・総括の空句**: 「重要なのは〜である」「本記事では〜を扱う／見ていく」「まとめると」「要するに」（直前の言い換えだけのとき）、「〜に他ならない」
- **「正面から」系**: 「正面から扱う」「正面から向き合う」など、中身の代わりに姿勢だけを宣言する言い回し
- **空虚な形容**: 「不可欠」「核心的」「鍵となる」「根本的な」「多角的」「包括的」「総合的」（何をどう見たかを書かず強調だけする）
- **空虚な動詞**: 「掘り下げる」「深掘りする」「触れる」「言及する」（何をどう書いたかを示さず終わる）
- **接続の型**: 「〜において」「〜の観点から」「〜という側面から」（新情報なし）、「さらに」「また」「加えて」の連打
- **空虚な強調**: 「非常に」「極めて」「大いに」（中身のない強調）

このリポの文体との両立（過剰反応の防止）:
- ヘッジ（「かもしれません」「思います」「気がします」）は本リポの文体資産。削らない。禁じるのは「根拠なく主張を弱める」緩和だけで、推量・保留・読者の疑念・括弧書き自己訂正としてのヘッジは保持する（ルール1・2と両立）
- 「つまり」「一方」「ただし」「ということで」などリズムを作る遷移句は対象外（ルール6で許容）
- 「言語化」はブログの目的そのもの（ルール15）なので、空虚な動詞としては扱わない

## 記事の典型構造

```
冒頭: 執筆動機から書き出す（AI が brief/thinking の動機をもとに下書き。一次感情の生の部分のみ ✍️ で著者が上書き） ／ 実験・技術共有系は圧縮した「何を書くか」宣言から入ってもよい
      ↓
（必要なら）参考リンク（短く）
      ↓
今回書くことの宣言（1-2文）＋ 執筆動機の共有（なぜ書くか・なぜやっているか／概要）
      ↓
（必要なら）## 概念導入（「簡単に」「自分の理解」付き）
      ↓
## 大セクション1（問い / 観察 / シンプルな宣言）
      ↓
## 大セクション2（対比・別側面）
   ### 余談：...（本筋に組み込めない時のみ）
      ↓
## 大セクション3（射程拡張: 個人→組織等）
   ### 価値の再評価（「否定→ただし」パターン、ただし用途限定）
      ↓
## 最後に
   - 型は記事の主張に応じて選ぶ（標準型 / 箇条書き集約型 / 問いかけ型 / 短い宣言型）
   - 末尾サインオフ:「質問などありましたら、XかLinkedinまでお願いします。それでは！」
```

大見出し (`##`) は3-4個に収まるよう、近接論点は1章にまとめる。

## 文章生成時のチェックリスト

`/write` でドラフトを書く時、以下を満たしているか確認:

### 基本トーン

- [ ] 一人称は「自分」になっているか
- [ ] 「〜と思います」「〜のように思います」等のヘッジを適度に入れたか
- [ ] **ヘッジ語尾の連続を避け、共感誘発の語尾 (ですよね、面白かったです 等) や括弧書き自己訂正と混在させたか**
- [ ] 強断定（「絶対」「必ず」「〜である」）を使っていないか
- [ ] 読者への問いかけ・呼びかけを1-2箇所入れたか
- [ ] 1段落2-4文・1トピックを維持しているか

### 強調・引用

- [ ] 重要フレーズ・概念に「」を付けたか
- [ ] **鉤括弧「」を1段落あたり1-2個以下に抑えているか** (念のための保険使用を避けたか)
- [ ] **太字 (`**`) を記事本文で使っていないか** (強調・マーキングは「」で行う)

### 構造

- [ ] **大見出し (##) は3-4個以内に収まっているか** (論点ごとに章を立てていないか)
- [ ] **章タイトルが「問い」「観察」「シンプルな宣言」の形で簡潔になっているか** (論点要約型・説明的に長い見出しを避けたか)
- [ ] **見出しで節の結論（オチ）を言い切っていないか** (見出しは問い・観察・対象にとどめたか)
- [ ] **抽象メタファーが章をまたいで繰り返し参照されていないか**
- [ ] **章間のロードマップ的遷移文 ("ここまでは〜" "両側から〜") を頻発させていないか**
- [ ] 「つまり」「一方」「ただし」「ということで」等の遷移句を入れたか
- [ ] **冒頭が体験・現在地、または圧縮した「何を書くか」宣言から始まっているか** (参考リンク先出し・年表/スコープ整理で導入を重くしていないか)
- [ ] **導入で執筆動機（なぜ書くか / なぜやっているか）を簡潔に共有したか** (「何を書くか」の宣言とは別に)

### AI 口調・空句

- [ ] **LLM 的な空句（「重要なのは〜」「正面から〜」「不可欠」「核心的」「掘り下げる」「〜の観点から」「非常に」等）を使っていないか** (ルール22)
- [ ] **空句を消した結果、根拠のあるヘッジ（思います・かもしれません）まで削っていないか** (ルール1・2のヘッジは残す)

### 内容の方向性

- [ ] 抽象論だけで終わらず、具体例・実体験を添えたか
- [ ] **順序が「具体観察 → 抽象解釈」になっているか** (抽象先行になっていないか)
- [ ] **製品/外側の批判ではなく、我々ユーザー側の主体性に話が向いているか**
- [ ] **現地体験 / 固有名詞 / セッション名 / タイムスタンプを具体的に出しているか**

### 表記・用語

- [ ] **表記辞書に従っているか** (例: 「AIエージェント」でなく「AI Agent」)
- [ ] **和文と英単語の境界に半角スペースを入れていないか** (「Tableau は」でなく「Tableauは」。英語フレーズ内部の語間スペースは残す)
- [ ] **数量・順番を算用数字で書いているか** (「二つ」でなく「2つ」。ただし「一方」「一貫」等の熟語内漢数字は残す)
- [ ] **ダッシュ（—／―／――）・中黒（・）を地の文・見出しで使っていないか** (並列は読点、補足・同格は括弧で)

### 結びと余談

- [ ] **「最後に」が標準型を機械的に踏襲していないか** (記事の主張に応じた畳み方を選んだか)
- [ ] サインオフ「質問などありましたら、XかLinkedinまでお願いします。それでは！」で締めたか
- [ ] **余談を入れる場合、本筋に組み込めなかったかを確認したか** (著者文体らしさのための余談化を避けたか)

### 冒頭の例外

- [ ] 冒頭は執筆動機から書き出し、AI が brief/thinking の動機をもとに下書きしたか（著者の手動追加を待たない／著者は後から修正）
- [ ] 一次感情・私的体験など AI が一次情報を持たない部分のみ、HTMLコメントで隠さず MDプレビューで見える ✍️ 可視マーカー（引用ブロック）として残したか

# 初見読者シミュレーション

AI が生成した記事ドラフトを **初見の読者として通読** し、読中体験・読後の反応・行動変化・炎上経路の予測を `output/{yyyymmdd}_$ARGUMENTS/readers/readers_NN.md` に出力する。

**フォルダ命名規則**: 出力先は `output/{yyyymmdd}_$ARGUMENTS/` とする。既存フォルダがあればそれを使う。`$ARGUMENTS` に日付プレフィックス付きのフォルダ名が直接渡された場合はそのまま使う。以降のパス記述で `$OUTPUT_DIR` はこのディレクトリを指す。

このスキルは隔離コンテキスト（`context: fork`）で実行される。会話履歴にはアクセスできないため、対象はすべてファイルから読む。書いた当人から切り離された別エージェントが「公開された記事」として読むことで、執筆時の意図や経緯を知らない読者の体験を近似する。メイン会話には**実行サマリのみ**を返す。指摘の全文を戻り値に詰めない。

## このスキルの役割

ドリフト点検や文体チェックでは出ない種類の問題——どこで読者が離脱するか、どの一文が刺さるか、どの段落が切り取られて燃えるか、読後に行動が変わるか——を、公開前に予測する専任。

工程内での役割分担を守る（重複させない）:

- **critique**: thinking.md の主張そのものの是非を叩く。本スキルは主張の是非を問わず、**読者にどう受け取られるか**だけを見る
- **check-drift**: 前段からの背骨落ち・無根拠 add を見る。本スキルは前段ファイルと照合しない（初見読者は thinking.md を知らない）
- **review-author-draft**: 著者が書いた draft_user.md の編集レビュー。本スキルは **AI が生成したドラフト**が対象

ここで出た指摘は推敲の材料であり、**ドラフトを書き換えない**。指摘の採否はメイン側（write スキルの採否基準）が判断する。

## 手順

### 1. 引数とテーマの確定

`$ARGUMENTS` を解釈する: 末尾トークンが `.md` で終わるならそれを対象ファイル名に採り、残りをテーマ名にする。対象ファイル名の既定は `draft.md`。

### 2. 入力の読み込み

- **対象ドラフト**（必須）: `$OUTPUT_DIR/{対象ファイル名}`。無い・空の場合は「対象が見つからないので、先に `/write` を実行してください」という旨をサマリで返して終了する
- **brief.md**（あれば）: 「読者像」と「コアメッセージ」だけをペルソナ設計に使う。それ以外の節（書く動機・観点一覧）は読まない——初見読者は著者の意図を知らないため
- `$OUTPUT_DIR/readers/` 内の既存 `readers_*.md`（あれば。前回指摘の解消／未解消を把握する）

**読者には見えない要素を除外する**: ドラフト冒頭の作業メモ（ℹ️ 引用ブロック）、✍️ 可視マーカー、HTML コメント（画像・リンクのプレースホルダー）は著者向けであり、読者体験の評価対象から外す。✍️ 部分は「ここに著者の一次体験が入る前提」で読む。

### 3. ペルソナの設計

3つのペルソナを立てる:

1. **主読者**: brief.md の「読者像」をそのまま使う（brief.md が無ければ記事本文から想定読者を推定する）
2. **懐疑側の読者**: 記事の主張に乗らない合理的な読者。記事が想定する反論を実際に抱えている人
3. **前向き側の読者**: すでに主張に共感している読者。シェア・引用するかどうかの目線

記事のテーマに応じて 2・3 は具体化してよい（例: 特定コミュニティの中堅ユーザー、同業の実務者）。誰を立てたかは出力に明記する。

### 4. シミュレーションの実行

各ペルソナで通読し、次を記録する:

- **読中体験**: どこで引き込まれ、どこで離脱しそうになり、どこで反発・疑問を感じたか（該当箇所の引用つき）
- **読後の一言**: その読者が SNS に書きそうなポスト文面を1つ想像する
- **行動変化の予測**: 読後に行動が変わるか。変わらないなら何が障壁か
- **コメント欄の予測**: 書かれそうな指摘・質問・反論

ペルソナを離れた**編集者視点**で、次をまとめる:

- 記事の最大の強み1つ（推敲で削られないよう明示する）
- 直すと効果が大きい弱点を優先度順に最大5つ（該当箇所の引用＋直し方の方向性つき）
- 炎上・誤読リスク: 切り取り引用されたとき別の意味になる段落、立場の違う読者に攻撃と映る表現

**規律**: 良い点を水増ししない。忖度なしで書く。ただし指摘には必ず該当箇所の引用を付け、印象論だけの指摘にしない。

### 5. 連番の決定と readers_NN.md の生成

`$OUTPUT_DIR/readers/` 内の既存 `readers_NN.md` の最大番号 + 1 を今回の番号とする（2桁ゼロ埋め）。1つも無ければ `readers_01.md`。フォルダが無ければ作成する。

```markdown
# 読者シミュレーション #{NN}（{yyyymmdd}）

## 対象
{対象ファイル名}（ペルソナ: {3者の一覧}）

## 前回からの変化
{readers_01 等が既にある場合、前回指摘の解消／未解消。初回は「初回」}

## ペルソナ別の読書体験

### 1. {主読者の名前}
- 引き込まれた箇所: {引用＋理由}
- 離脱・反発した箇所: {引用＋理由}
- 読後の一言（SNS想定）: {ポスト文面}
- 行動変化: {予測と根拠}
- コメント欄の予測: {箇条書き}

### 2. {懐疑側} / ### 3. {前向き側}
{同構成}

## 編集者視点

### 最大の強み（削らないこと）
{1つ、引用つき}

### 直すと効果が大きい弱点（優先度順・最大5つ）
1. **{見出し}** — {引用} / {直し方の方向性}

### 炎上・誤読リスク
{切り取り・誤読の経路と緩和の方向性。無ければ「なし」}
```

`{yyyymmdd}` は実行日の日付。

### 6. 実行サマリを返す

メイン会話には、著者がファイルを開かずに要点を判断できる箇条書きサマリを返す:

```
対象: {対象ファイル名}／ファイル: readers/readers_NN.md

最大の強み: {1行}
弱点（優先度順・最大5件）: 各1行（見出し＋一言）
炎上・誤読リスク: {1行、無ければ「なし」}
行動変化の予測: ペルソナごとに1行
```

Write に失敗した場合は inline で指摘本文を返さず、「シミュレーションは実行したが保存に失敗」とサマリで明示する。

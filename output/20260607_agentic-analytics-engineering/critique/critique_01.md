## 指摘

### 1. dbt ベンチの数値が、Tableau の主張をどこまで支えられるか
- レンズ: 論理の飛躍 / 前提の穴
- 問題: 観点1の論理の背骨は dbt の「semantic layer 98–100% vs text-to-SQL 84–90%」という数値。だが thinking.md 自身が「Tableau の PDS は MetricFlow のような決定的コンパイルを持たない」と認めている。つまり数値を出した semantic layer（dbt の MetricFlow）と、記事が整備する対象（Tableau PDS のメタデータ）は別物。「dbt のコンセプトを借りる」と線引きしても、**数値の説得力は決定的コンパイルに由来する**のだとしたら、PDS のメタデータ整備にその数値を援用すること自体が飛躍になりうる。読者（特に懐疑的な実務者）は「その 98% は Tableau では出ないのでは？」と突く。
- 次の一手: 次ループで「PDS のメタデータ整備が text-to-SQL の精度を上げる」ことの Tableau 固有の根拠を一つでも持てるか掘る。dbt の数値を「精度の保証」ではなく「方向性のアナロジー」として使うのか、それとも著者の手元実験で何か観測があるのか、provenance を分けて書き分ける。なければ「数値は dbt 文脈の参考値」と明示的に格下げする判断を固める。

### 2. 「GUI ゆえ手作業が辛い」は本当に AI でなければ埋まらないギャップか
- レンズ: 想定反論
- 問題: 観点2のモチベーションは「GUI 製品で手作業が辛い → だから AI エージェント」。だが thinking.md 自身が「REST API / Metadata API / VizQL Data Service を備える」と書いている。API があるなら、従来型のスクリプト・決定的な変換ツールでも自動化はできる。「なぜ決定的な自動化ではなく AI エージェントなのか」の差分が弱い。AI が要るのは「解析・分解の判断」部分のはずだが、そこは"良さげに分解する"という非決定的な作業で、parity 検証で担保するという構図。読者は「結局 AI の分解判断は信頼できるのか／決定的ツールで十分な領域まで AI に投げていないか」と返す。
- 次の一手: 「AI でないと辛い部分（層の切り方の判断・メタデータ草案の自然言語生成）」と「API/スクリプトで十分な部分（DL・publish・行数比較）」の境界を一段切り分ける。AI の旨味が出る領域を具体的に名指しすると、観点2の主張が締まる。

### 3. 二軸の書き分けが「飛躍を消す」と言うが、data modeling 軸自体に飛躍が残る
- レンズ: 論理の飛躍
- 問題: 「semantic layer＝直接、data modeling＝追跡可能性＋精度」と分けたことで Prep 分解→agentic の飛躍が消えた、と気づきに書いている。だが data modeling 軸の「精度にも効く」根拠も dbt ベンチ（「最小 modeling 追加で精度改善」）一点に依存。しかも Tableau Prep の層化＝staging/intermediate/marts は dbt の SQL モデル分割とは実装が違う（Prep 出力は Hyper Extract で全行マテリアライズ、と brief にある）。dbt の「modeling 追加で精度改善」が、Prep を層化したときに同じ機序で効く保証はない。「追跡可能性」は構造の話なので転用しやすいが、「精度」の転用は #1 と同じ飛躍を抱える。
- 次の一手: data modeling 軸を「追跡可能性（lineage/explainability）」を主、「精度」を従（dbt 参考値・要検証）に重みづけし直すと、論理が崩れにくい。「精度」を主張の主柱に据えるなら手元での観測を一つ要する。

### 4. コアメッセージが「公式と同じ船」に寄りかかると、記事独自の主張が薄まる
- レンズ: コアメッセージの弱さ
- 問題: コアメッセージは「公式の潮流（agentic analytics / Composable Data Sources）と同じ船で、既存資産を未来に耐える形へ作り替える前工程」と位置づける。だが Composable Data Sources が「fct/dim/rpt の物理 JOIN 回避策を公式が肩代わりする方向」（brief 49行）なら、**公式が追いついた時点でこの実験の価値の一部は陳腐化する**。「同じ船」を強調するほど「じゃあ公式を待てばいいのでは」という反論を自分で招く。読後に残るのが「公式もやってる」だと、著者独自の見立ての輪郭がぼやける。
- 次の一手: 「公式が来るまでの transitional な解」なのか「公式が来ても残る価値（既存 Prep 資産の解析・既存ロジックの層化という公式が肩代わりしない部分）」なのか、立ち位置を一つに決める。後者を主張できると記事の賞味期限が延びる。brief 71行で著者自身がこの論点を「要検討」に挙げており、ここを詰めるのが最優先。

### 5. 動機の二重性（共有 ＋ 工学的興味）が、記事の焦点をぼかすリスク
- レンズ: コアメッセージの弱さ / 前提の穴
- 問題: 「主動機はライトな共有」「加えて AI に Prep を作らせること自体への工学的興味（半独立の動機）」と二つ持つ。実態に誠実なのは良いが、読者に向けた記事としては「結局この記事は何を読者に持ち帰らせたいのか」が二焦点になる。半独立の動機は「別記事候補」に退避しているが、書いている最中に工学的興味が漏れ出すと、読者像A（API を叩かない実務者）には how が重すぎる方向にぶれる。
- 次の一手: 「読後の読者の状態」（thinking.md 13–17行）は既に明確なので、工学的興味は本記事では完全に背景に退け、漏れたら別記事の予告として一行に留める線を引く。動機の二重性は thinking では持ちつつ、記事のメッセージは一焦点に絞る。

### 6. 読者像と内容の難易度ギャップ（前提知識の量）
- レンズ: 前提の穴
- 問題: 読者像A は「dbt / analytics engineering / semantic layer / MCP / text-to-SQL に必ずしも明るくない」（brief 17行）。一方コアメッセージは semantic layer・data modeling・text-to-SQL・MCP・lineage・Composable Data Sources を全部前提に組み立てている。「軽く導入が要る」とあるが、これだけの概念を軽い導入で繋ぐと、共有という主目的に対して認知負荷が高すぎる恐れ。主張の妥当性ではなく「届くか」の前提が崩れうる。
- 次の一手: critique の領分（主張の妥当性）からは外れるが、deepen/outline に渡す論点として: コアメッセージを「読者が一つだけ持ち帰る命題」に圧縮できるか試す（例:「Tableau の勝負どころは可視化からデータソースの質に移った」一点）。概念の導入順は outline の仕事。

## 総評
最優先は **指摘#1 と #3（dbt ベンチ数値の Tableau への転用が、主張の論理的背骨でありながら最も飛躍が大きい）**。「精度」の援用を方向性のアナロジーに格下げし、「追跡可能性」を data modeling 軸の主柱に据え直すと、記事全体の論理が崩れにくくなる。次点は **#4（公式と同じ船に寄りかかると独自価値が陳腐化する論点）**——著者自身が brief で「要検討」に挙げており、ここを決め切ると記事の輪郭と賞味期限が定まる。

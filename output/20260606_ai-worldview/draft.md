この1年間でAI Agent系の話題に多く触れてきました。この記事では1年間の学びを、TableauユーザーかつAI Agentトピックにそこまで触れてこなかった方に、「Tableau MCPやAgentic Analyticsを語る際には、この辺りを基準にすると良いですよ」という内容をお伝えする試みを行います。

技術の話は深く行わず、あくまでも概要レベルに留めます。

この記事で目指すことは、読んだ方が以下の議論をしなくなることです。

- 「Tableau MCPを使えばAIはデータ分析が出来るの？」
- 「AIを使ったデータ分析は正しいの？」

どういう議論をするようになって欲しいのか？と言えば、例えば以下です。

- 「AIにデータ分析をさせたり、AIを使って人間のデータ利活用を促進するためには、どのように設計をすればいいのか？」
- 「いまAIに与えられているドメイン知識や業務知識は何で、何を与えられていないのか？それは定型化できるのか？」

要は「AIに何が出来るか」「MCPで何が出来るか」という軸足から、「AIに自分がやってほしいことを実行してもらうために、何を自分が用意・整備するべきか」という軸足に移ってほしい、それをTableauコミュニティの共通言語にしてほしい、という願いです。

## AIの分析は、正確だった？

冒頭で「やめたい議論」として、「AIを使ったデータ分析は正しいの？」を挙げました。まずはここから考えてみたいと思います。正直に言うと、自分自身もこの問いで会話をしてきた側です。

ただ、最近少し考えが変わってきました。AIに集計や分析を任せて結果が外れたとき、まず疑うべきはモデルの賢さなのだろうか、と思うようになったんです。多くの場合、それは「設計」の問題なのではないか、という気がしています。

ここで言う設計を、ひとまず「AIにどう動いてほしいかを、こちらがどれだけ示せているか」と捉えてみてください。指標の定義、集計の粒度、どの手順で分析してほしいか。そういうものを渡さないまま「正しく分析できるか」を問うのは、AIの能力ではなく、自分の設計を試していることになるのかもしれません。

もちろん、これは使っているモデルにもよります。ただ、ChatGPT や Claude、Gemini といった今のモデルを使っている範囲で言えば、「うまくいかない」の多くは能力ではなく設計に原因がある、と疑ってみる価値はあると思っています。「チャットに繋いで試す」から「設計した上で使う」へ。記事を通して動かしたいのは、この基準点です。

## ハーネスという基準点

この半年ほどで、AIエージェントまわりの議論はずいぶん成熟したように見えます。スキル、ハーネス、コンテキスト設計といった言葉が、いつの間にか標準語彙になりました。例えばスキルやコンテキスト設計の話は、Anthropic の[Agent Skills](https://www.anthropic.com/news/skills)や[コンテキストエンジニアリングの記事](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)あたりに、その空気がよく出ている気がします。

一方で、Tableau コミュニティの会話には、この前提がまだ十分には入りきっていない気がします。「AIにTableauのデータをそのままつないだらどうなるのか」という話で止まっていることが多い。自分が埋めたいのは、この段差です。

ここで言う「ハーネス」を、自分の理解で簡単に説明すると、AIがタスクを実行するための足場のことです。福島良典さんは[エージェントハーネスの記事](https://note.com/fukkyy/n/n1d8fce44e67a)の中で、AIエージェントを「シェフ」、ハーネスを「調理場」に喩えていました。腕のいいシェフでも、調理場が整っていなければ良い料理は出てきません。コンテキストや手順、スキルを設計して、AIから見える「盤面」を閉じてあげる。そうやって、長く動かしたときに起きる手順の取りこぼしや品質の思い込み（記事では「ドリフト」と呼ばれていました）を抑えていく営みだと捉えています。

では、設計が足りているのかどうかは、どう見分ければいいのでしょうか。一つ分かりやすいのは、集計結果を突き合わせてみることだと思います。AIが出した数字がおかしいと感じたら、同じ集計を SQL や VDS（VizQL Data Service）のような決定論的なツールで計算し直して比べてみる。そこでズレるなら、それは計算そのものをAIに任せてしまっている設計の問題です。ツールで出した正しい数字をAIが取り違えるようになって、初めて「能力の限界かもしれない」と疑えばいい。この順番で見ると、「設計を疑え」がただの精神論ではなく、原因を切り分ける手順になってくれます。

## 脳・手・環境という地図

もう少し踏み込むと、生成AIの活用は「脳」「手」「環境」の設計でほぼ決まる、というのが自分の今の理解です。厳密な分類というより、自分の構成を点検するための地図だと思ってもらえればと思います。

- **脳**：LLM そのものに加えて、スキルやドメイン知識、ワークフロー手順を書いたテキスト群。何を・なぜ・どの手順でやるか、を持っている部分です。
- **手**：ツールやスクリプト。与えられたタスクを実際に実行する手段を持っているか、という部分です。
- **環境**：サンドボックスや、ツールの権限まわり。どこで・どの権限で動くか、という部分です。

この三つで、「ChatGPT に Tableau MCP を繋いだだけ」の状態を当ててみると、なかなかスカスカに見えてこないでしょうか。脳はプロンプト程度、手は MCP だけ、環境はほぼ既定のまま。これで「正しく分析できるか」を問うのは、地図のほとんどが空白のまま走り出しているようなものかもしれません。Agentic な分析を本当に実現しようとすると、この各軸に、それなりに広い理解と実践が要るのだと思います。

そして、この地図は点検にも使えます。冒頭で書いた「いまAIに、どんなドメイン知識や業務知識を渡せていて、何が渡せていないのか。それは定型化できるのか」という問いは、まさに脳の部分を見ているわけです。「AIに何ができるか」ではなく「自分が何を脳・手・環境に用意できているか」へ。軸足を移すというのは、こういうことなのだと思います。

この分け方は、自分が勝手に言っているだけではなくて、Anthropic の[Scaling Managed Agents](https://www.anthropic.com/engineering/managed-agents)の考え方とも地続きだと感じています。あの記事では、アーキテクチャを「脳（Claude とハーネス）」と「手（サンドボックスとツール）」に分けていました。自分は手の側を、実行手段としての「手」と、実行の場としての「環境」にもう一段割っているだけで、本質は同じだと捉えています（細かい話ですが、自分の言う環境は認証が済んだ前提での権限スコープを指していて、認証そのものは設計の前提条件として外に置いています）。

抽象的な話が続いたので、実際に動かしたものに触れておきます。先日 Tableau Now! で登壇した際に、Tableau MCP と Coding Agent を組み合わせて、分析からレポート生成までをAIに任せる構成を紹介しました。リポジトリの中身は、分析手順を入れた `.agents/skills`、共通ルールを置いた `rules-bank`、目次役の `AGENTS.md`、といった作りになっています。これがまさに、脳・手・環境を設計したハーネスの実体です。

## 最後に

この記事で一番動かしたかったのは、Tableau MCP の評価のしかたです。Tableau MCP は、結局のところ「Tableau のコンテンツを使うための道具」です。だとすれば、「この道具自体が分析に役立つのか」を試して終わるのは、少しもったいない気がします。問いたいのはむしろ、「この道具を使って、自分たちのデータ利活用にどんな価値を取り込むか。そのために、どんな設計をするか」の方ではないでしょうか。

そう考えると、AIに任せられる範囲が広がるほど、人間が考えるべきことはむしろ「設計」に寄っていくのだと思います。OpenAI の Harness engineering の記事に、「Humans steer. Agents execute.（人間が舵を取り、エージェントが実行する）」という一節がありました。エンジニアの仕事が、コードを書くことから、環境を設計し・意図を示し・フィードバックの仕組みを作ることへ移っていく、と。脳・手・環境をどう組むかを決めるのは、やはり人間の側なんですよね。（ちなみに、「そもそも何を分析するか」「出た示唆を業務の判断にどう変えるか」も人間に残る仕事ですが、これは設計とはまた別の話なので、別の記事で書いてみたいと思います。）

とはいえ、いきなりフルのハーネスを組もうとしなくていいと思っています。今チャットに繋いで分析させている人が明日できる最小の一歩は、たぶん「指標の定義や粒度、用語集をテキスト一枚にまとめて渡す」ことです。それだけでも、指標の取り違えや粒度のズレはかなり減るはずです。これがハーネスの最小単位であり、設計の第一歩だと思います。まずは一枚から、で十分ではないでしょうか。

### 余談：この記事も同じやり方で書きました

正直に白状すると、この記事も似たような進め方で書いています。考えを掘る工程、それを批判的にレビューする工程、最後に自分の判断が要るところだけ自分に戻す工程、というふうに分けて、AIに手伝ってもらいました。ただ、これは「だから良い記事になった」という証拠の話ではありません。あくまで、同じことが皆さんの業務でもできるはず、という地続きの実例として置いておくだけです。

各軸（脳・手・環境）をどこまで作り込むか、どこが自分のドメインにとっての勘所か、というのは、結局は手を動かしてみないと見えてこない部分だと思います。自分もまだ探っている最中なので、また何か分かったら、その時に書いてみます。

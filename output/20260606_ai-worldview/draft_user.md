# 1年間AI Agentに触れてきたTableauユーザーが思う、Tableau MCPとAgentic Analyticsとの付き合い方

公開日:	2026/06/06
最終更新:	2026/06/06

参考資料

この1年間でAI Agent系の話題に多く触れてきました。この記事では1年間の学びを、TableauユーザーかつAI Agentトピックにそこまで触れてこなかった方に、「Tableau MCPやAgentic Analyticsを語る際には、この辺りを基準にすると良いですよ」という内容をお伝えする試みを行います。

技術の話は深く行わず、あくまでも概要レベルに留めます。
この記事で目指すことは、読んだ方が以下の議論をしなくなることです。

- Tableau MCPを使えばAIはデータ分析が出来るの？
- AIを使ったデータ分析は正しいの？

どういう議論をするようになって欲しいのか？と言えば、例えば以下です。

- AIにデータ分析をさせたり、AIを使って人間のデータ利活用を促進するためには、どのように設計をすればいいのか？
- いまAIに与えられているドメイン知識や業務知識は何で、何を与えられていないのか？それは定型化できるのか？

要は「AIに何が出来るか」「MCPで何が出来るか」という軸足から、「AIに自分がやってほしいことを実行してもらうために、何を自分が用意・整備するべきか」という軸足に移ってほしい、それをTableauコミュニティの共通言語にしてほしい、という願いです。

## AI Agentを「脳」と「手」で考える

この章はAnthropicのScaling Managed Agentsの内容を大いに参考にしています。

（2026年6月初頭から）ここ半年強ほどで、AIエージェントまわりの議論は「ハーネスをどう整えるか」という話が標準になってきた気がします。広い意味でのハーネスエンジニアリング、つまりAIに仕事をさせるための足場をどう設計するか、という話です。

ちなみに「広い意味で」と書いた理由は、例えばこちらの記事を自分は参考にしているためです。
AIエージェントの″ハーネス″に関わる混乱と私見
（まぁOpenAIもAnthropicも、対外向け発信では細かく区別していない印象もありますが...例えばこちらのOpenAIの記事）

例えばAnthropicを見てみると、ざっくりと以下の時系列で記事が公開されています。

- 2025年9月: Effective context engineering for AI agents
- 2025年10月: Equipping agents for the real world with Agent Skills
- 2025年11月: Effective harnesses for long-running agents
- 2026年3月: Harness design for long-running application development
- 2026年4月: Scaling Managed Agents: Decoupling the brain from the hands

コンテキストエンジニアリング → Skills → ハーネスエンジニアリングと、言葉こそ変わりつつありますが、AI Agentに目的を達成してもらうための設計に関する議論が主になっています。

冒頭のScaling Managed Agents記事では、AIエージェントのアーキテクチャを「脳（LLMとハーネス）」と「手（実行環境とツール）」に分けて捉えていました。ハーネスはこの「脳」の側にあたる、AIに何を・どの手順でやらせるかに関する情報です。

上記の記事ではClaudeの狭義/内部ハーネスの話が続きますが、Skillsやプロンプト等も結局はLLMに入力されるので、ここでは「脳（LLMと内外両方のハーネス）」として扱います。

この「脳」と「手」について、少しTableau文脈での具体例を挙げてみます。
以降、簡単のために脳＝ハーネス、手＝ツールで考えます。

- 脳（ハーネス）：こちらが用意する知識やルールなど
  - 指標・KPIの定義（売上の計算方法、集計の粒度、対象期間の取り方）
  - 業務・ドメイン知識（自社の用語集、事業の文脈、いつも見ている観点）
  - 分析手順をまとめたドキュメント
  - 命名規則やレポートの体裁といったルール
- 手（ツール）：タスクを実行するための手段
  - Tableau MCP（Tableau のコンテンツやメタデータに触る）
  - Tableau REST API といった各種 API
  - 集計や検証のためのスクリプト（SQL や Python など）

この「脳」と「手」で考えたときに、冒頭の2つの問いの問題点が見えてきます。
次の章で細かく見ていきましょう。

- Tableau MCPを使えばAIはデータ分析が出来るの？
- AIを使ったデータ分析は正しいの？

## Tableau MCPだけ・Tableau MCP + チャットAIだけで論じないために

前者の問いは「手」の話しかしていません。

Tableauのデータソース等を使うためのツールをTableau MCPが提供しているだけなので、「何らかの集計作業が出来る」などの機能要件は満たせますが、「どのように集計するべきか」「データを使って何をするべきなのか」については、完全に宙に浮いています。

先ほどの「脳」と「手」に当ててみると、脳はプロンプト程度、手は MCP だけ。なかなかスカスカに見えてこないでしょうか。これで「分析が出来るか」を問うのは、地図のほとんどが空白のまま走り出しているようなものかもしれません。

言い換えると、いま自分のAIに、どんなドメイン知識や業務知識を渡せていて、何が渡せていないのか。それは定型化できるのか。冒頭で書いたこの問いは、まさに脳の部分を点検しています。

後者の問いは「脳」をどう整えるか、という観点がありません（LLMの素の頭脳とプロンプトだけに頼る状況）。

そもそも「正しい」と言うためには、何をもって正しいとするかが先に決まっている必要がありますよね。正しい集計方法とは何か、正しい論理や手順とは何か、どのように正しさを担保するか、etc...

例えば正しい集計方法の情報を与えるためには、使うべきデータやクエリを事前定義し連携しておく、など。
計算の設計そのものをAIに完全に任せてるのではなく、一定の方針を「脳」に与えておき、また検算の方法を「手」に与えておく、などの設計に関する思考があるべきだと思います。

ここまでの議論が、自分が冒頭で「AIに自分がやってほしいことを実行してもらうために、何を自分が用意・整備するべきか」に思考を向けるべきと書いた理由です。

「脳」と「手」をしっかり整えた上でのAI活用を議論するべきであり、手だけを切り出した議論やデモで終わらせず、よりAgenticな話をした方が良いと思いますという提言でした。

著者注：意識的に独立して扱っているのであれば、Tableau MCP単体のデモも良いと思うんです。一方でTableau MCP単体のデモや実験それだけで「AIはTableauデータを使える／使えない」というような議論になることは良くないと思います、という話です。

## 最後に

Tableau は「Agentic Analytics Platform」になるんだと、TC26にて打ち出しました。Agentic Analytics という言葉も、一気に身近になってきましたよね。

同時にTableauユーザーは「Architect」になるんだ、という話もしました。その中のひとつが「Agentic Architect」であり、プラットフォームやAIツールは何であれ、AI Agent（が動きやすい状態）を作る人というロールを、自分含めたTableauユーザーが担う方向になるのかなと。

（関連：Tableau Conference 2026に参加した感想）

このような流れが予想される中で、「現在のAI Agentトピックにおける、重要かつ基本にしたい考え方は何か」のエッセンスを、微力ながらお伝えしたいことが、本記事の動機でした。

Agentic Analytics は、「AIが勝手に分析してくれる」ことではなく、「自分が脳と手を設計して、AIに分析してもらう」ことだと思っています。

プラットフォームが Agentic になっていくほど、私たちユーザーの仕事は「設計」の側に寄っていく。Tableau MCP も、その設計に使う道具の一つにすぎません。

（何ならTableau MCPに限定せず、Tableauの各種APIを直接AI Agentに使わせても良いですし）

「AIに何ができるか」「MCPで何ができるか」ではなく、「AIに自分のやってほしいことをしてもらうために、自分は何を用意・整備するべきか」。この軸足が、Tableauコミュニティの共通言語になったら嬉しいなと思っています。

脳と手をどこまで作り込むか、どこが自分のドメインにとっての勘所か、というのは、結局は手を動かしてみないと見えてこない部分だと思います。
自分もまだ探っている最中なので、皆さんと学びながら一緒にAgentic Analytics時代を楽しんでいけたらと思います。まぁ実際に、この話はとても楽しいのですが！

## 余談

個人的には「脳」と「手」と「環境」の3つで考える方が腹落ちしています

- ここでの「環境」とは、Anthropicの整理の中にあった「サンドボックス」の話と、例えばTableauのパーミッションの話などの、ツールを使うにあたって必要な外部環境は揃っているかという観点です。
- 道具（の実装）だけあっても使えない、という場合があるため。
- 本文から技術的な話を除外したかったので、本記事では割愛しました。
- データガバナンスやセキュリティ観点で、この「環境」の話が一番ややこしくなる印象があるものの...
- まずはAI Agentの概念を掴みましょう、ということで...

（Tableau MCP含めて）一概にMCPが「手／ツール」というわけではない

- 実際には、MCP自体がツールの説明文、入出力メッセージなど「脳」にまたがる内容を持っています。
- この記事では簡単のためTableau MCPを手／ツールとして置きましたが、例えばTableau MCPのquery-datasource自体に「どう使うべきか」等の説明がしっかりあります。そしてLLMに入力されます。
- あとはMCPの実装によっては、参考情報をResourceとして渡す場合もある、など。何ならMCP Toolとして別の生成AIを呼ぶとかも可能です。
- ということで、一概にMCP＝手／ツール、という整理が出来なかったりもします。
- とはいえ、本記事では簡単のために上記の整理としました。

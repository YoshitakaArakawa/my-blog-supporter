# 「チャットに繋ぐ」から「ハーネスを組む」へ ― Tableau×生成AIで見えている世界を揃える（タイトル仮）

## テーマ
Tableau×生成AIの“今のスタンダード”は、チャットUIにMCPやデータを直結して「AIは正しく分析できるか」を試す段階ではない。コンテキストを整理し、スキルを整備し、「何のために・どの手順で分析するか」を定義した上でAIに作業させること――広義の**ハーネスエンジニアリング**――が前提になっている。ハーネスを整えたかどうかで、同じ「Tableau×生成AI」を見ていても見える世界が違う。

## 書く動機
- Tableau MCPを扱うユーザー会・ブログ記事が増えてきた。
- だがその多くが「チャットUIにTableau MCP/データを直接つないで、生成AIが正しく分析できるか試す」デモに留まっている。これは1.5年前のフレーム。
- 現在の主流は、コンテキスト整理・スキル整備・手順定義（＝ハーネス）を済ませた上でAIに作業させる形。この認識差を埋めたい。
- 自分が学んできた結果の共有・還元として書く。説教ではなく「学びの助け」のトーン。

## 読者像
生成AIの最前線（スキル、AIエージェント）を必ずしも追えていない**Tableauユーザー**。
- Tableauユーザー全員がこのステージにいるわけではない。一方で最前線をキャッチアップしている層も一定数いる。
- その両者が**同じ目線で会話できる機会を増やす**のが狙い。
- 「今の生成AI活用のスタンダードはハーネス込みで考えること」を提示し、先行層との会話の足場を作る。

## コアメッセージの方向性
生成AI活用は、モデルに何かを“入れて試す”ことではなく、**脳・手・環境の設計**である。
- **脳** = モデル（Claude等）＋ハーネス：何を・なぜ・どの手順で、を握る部分
- **手** = ツール（Tableau MCP、各種API、サンドボックス）：実際に作業を実行する部分
- **環境** = コンテキストとスキルを整えた作業空間＝ハーネス
この3つの設計が全て。ハーネスを整えた人と整えていない人では、見えている世界が違う。

**拡大解釈（背骨の言い換え）**: ハーネスエンジニアリングとは、**業務を「完全情報ゲーム」に寄せる営み**である。
- ChatUI+MCPは、ルール・目的・制約・手順・コンテキストがAIから見えない＝**非完全情報**。だからブレる（ドリフトする）。
- Skills／AGENTS.md／CLAUDE.md／コンテキスト設計で**盤面を閉じて見えるようにする＝完全情報ゲームに寄せる**。これがハーネスの正体。
- ただし業務の全てを完全情報化はできない。「何を分析すべきか・示唆をどう解釈するか」は本質的に盤外（非完全情報）＝**人間が握る「問いと判断」**。

## アイデア・観点の一覧

### 掴み（現状への違和感）
- Tableau MCPの登場で、ユーザー会・記事が一気に増えた。
- よく見るデモ：チャットUIにTableau MCP/生データを直接つなぎ、「生成AIは正しくデータ分析できるか？」を試す。
- これは1.5年前なら自然だった問い。今は問いの立て方が古い。

### 掴みの第2波（独立小節：精度論争への楔）
- 「生成AIの分析、正確だった/不正確だった」という議論をよく見かける。
- 私見：**結果が外れたとき、まず疑うべきはモデルの賢さより設計（ハーネス）**。集計・分析をAIに渡して失敗したら、能力ではなく設計を疑う方がいい。
- ＝この議論は「モデルの能力テスト」をしている。だが本当は「盤面（設計）の話」。1.5年前フレーム批判の最も具体的な現れ。背骨へ橋渡しする楔。
- **留保（反証不能化を避ける）**: 絶対化しない。能力起因の失敗（大量データの暗算、スキーマ誤読等）は存在する。正直な形は「frontierモデル（ChatGPT/Claude/Gemini級）を、決定論的計算をループに組み込んで使う前提なら、分析失敗の多くは設計起因 → まず設計を疑う価値がある」。
- **「設計」の具体**（空疎にしない）: ①数値はLLMに暗算させず決定論的ツール（SQL/VDS/コード）に計算させたか ②粒度・指標・フィルタの定義を与えたか ③スキーマ・意味定義を渡したか。→ 脳（手順定義）／手（ツールに計算させる）／環境（指標と意味の定義）の③に直結。
- 自分の実例：VDS API で行数・列を比較して成果物を検証する（＝設計で正確性を担保する）話。能力に頼らず仕組みで担保する具体。
- トーン：「お前の設計が悪い」ではなく「**まず設計を疑うと景色が変わる**」（学び還元）。

### 背骨（テーマ②：見えている世界が違う）
- 今の主流：コンテキストを整理 → スキルを整備 → 「何のために・どの手順で分析するか」を定義 → その上でAIに作業させる。
- これが広義の**ハーネスエンジニアリング**＝業務を完全情報ゲームに寄せること。
- ハーネスを整えたかどうかで、「Tableau×生成AI」の解像度・打ち手・期待値が根本的に変わる＝見える世界が違う。
- 「AIが正しく分析できるか」は、ハーネス次第で答えが変わる問い。モデル単体の性能テストではない。
- **3つのアプローチ（自身の発表 p19）**: 補助＝GPTs/RAG（AIに聞いてTableauに持ち帰る）／ 統合＝Extensions（Tableauの中でAIが動く）／ 自律＝MCP+エージェント（育てる）。コミュニティの議論は補助・統合に留まりがち。背骨は「自律」の世界を提示すること。

### 支柱（テーマ③：脳・手・環境）
- 生成AI活用＝脳・手・環境の設計が全て、という自分の見立て。
- Anthropic「Managed Agents」を引用：アーキテクチャの根本が**「脳（Claude＋ハーネス）と手（サンドボックス＋ツール）の分離」**。`The harness leaves the container`。
- 環境＝ハーネス＝コンテキストとスキルを整えた作業空間。ここの設計が成果を決める。
- Tableauの文脈に翻訳：Tableau MCPは“手”の一部にすぎない。脳（手順・目的の定義）と環境（コンテキスト・スキル整備）を欠いたら、手だけ立派でも分析は再現しない。
- **具体例（自身の発表 p16）**: `.agents/skills`（analysis-planning, reporting…）, `rules-bank`（common-rules.md, conversation-contract.md…）, `AGENTS.md` の実リポ構成＝Tableau分析エージェントのハーネスの実体。
- 福島良典（LayerX）の「**ドリフト問題**」（終了条件の誤認・品質の過信・累積的ズレ）＝非完全情報の症状。対処＝外部チェックリスト・視覚的検証・決定論的フック＝ハーネスで完全情報側に寄せる。

### 人間の役割（批判への応答。結びへ繋ぐ）
- 業務を完全情報に寄せるほど、価値は**盤面を定義する側＝ハーネス設計者**へ移動する。
- 人間が握るのは「**問いと判断**」（自身の発表 p17：示唆・次アクションはAI、問いと判断は人間）。
- **業務知識の深さがAI分析品質を決める**（p18：ドメインエキスパートこそ最大価値。技術スキルのギャップはAIが埋める）。
- ＝「ハーネス整えれば分析が自動化する」ではない。盤面を設計し、問いと判断を握る人になる、という話。

### 結び（テーマ④へ接続）
- だから「チャットに入れて試す」段階で止まらず、ハーネスを組む側＝盤面を設計する側に回ろう。
- 話を聞くより手を動かした方が早い：ハーネスは作って初めて世界が見える。
- 還元のトーン：自分が学んできた範囲の共有。一緒に同じ目線に立つための補助線。ChatUIデモは否定でなく“入口／前段”として置く。

### 余談候補（サイドストーリー・記事の最後の方に軽く）
- この記事自体を書くワークフロー（AI支援のブログ執筆harness）が、記事の主張の実物例になっている、という self-reference。
- 具体: 思考を掘る工程（deepen）に対し、脳・手・環境を設計し、批判レビュー（critique）を自走させつつ「著者固有の判断が要る論点だけ人間に戻す」content-typed gate を組んだ。＝「問いと判断は人間が握る」をワークフロー自身に適用した。
- メタな落とし: ハーネスを組む話を、ハーネスを組みながら書いた。読者にも「あなたの業務でも同じことができる」と地続きに見せる結び。
- 分量は軽く。余談として1段落程度。主軸を食わない。

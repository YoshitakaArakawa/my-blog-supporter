この1年間でAI Agent系の話題に多く触れてきました。この記事では1年間の学びを、TableauユーザーかつAI Agentトピックにそこまで触れてこなかった方に、「Tableau MCPやAgentic Analyticsを語る際には、この辺りを基準にすると良いですよ」という内容をお伝えする試みを行います。

技術の話は深く行わず、あくまでも概要レベルに留めます。

この記事で目指すことは、読んだ方が以下の議論をしなくなることです。

- 「Tableau MCPを使えばAIはデータ分析が出来るの？」
- 「AIを使ったデータ分析は正しいの？」

どういう議論をするようになって欲しいのか？と言えば、例えば以下です。

- 「AIにデータ分析をさせたり、AIを使って人間のデータ利活用を促進するためには、どのように設計をすればいいのか？」
- 「いまAIに与えられているドメイン知識や業務知識は何で、何を与えられていないのか？それは定型化できるのか？」

要は「AIに何が出来るか」「MCPで何が出来るか」という軸足から、「AIに自分がやってほしいことを実行してもらうために、何を自分が用意・整備するべきか」という軸足に移ってほしい、それをTableauコミュニティの共通言語にしてほしい、という願いです。

## 脳と手という考え方

ここ1年ほどで、AIエージェントまわりの議論は「ハーネスをどう整えるか」という話に中心が移ってきた気がします。広い意味でのハーネスエンジニアリング、つまりAIに仕事をさせるための足場をどう設計するか、という話です。スキルやコンテキスト設計といった言葉が標準語彙になってきたのも、Anthropic の[コンテキストエンジニアリングの記事](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)（2025年9月）や[Agent Skills](https://www.anthropic.com/news/skills)（2025年10月〜）あたりに、その空気が出ている気がします。そして最近は「ハーネス」という言い方そのもので語られることも増えてきました。

ここで「広い意味で」と書いたのには理由があります。ハーネスという言葉は、モデルを作る側が実装する内側のハーネス（狭義）と、使う側が整える外側のハーネス（広義）で、指しているものが違うんですよね。この整理は watany さんの「[ハーネスに関わる混乱と私見](https://zenn.dev/watany/articles/d8b692bbca65a3)」がわかりやすいです。この記事で扱いたいのは、我々 Tableau ユーザーのような使う側が整える、外側のハーネスの方です。

このハーネスを自分の理解で簡単に言うと、AIがタスクを実行するための、モデルの周りを固める足場のことです。Anthropic の[Scaling Managed Agents](https://www.anthropic.com/engineering/managed-agents)の記事では、AIエージェントのアーキテクチャを「脳（Claude とハーネス）」と「手（サンドボックスとツール）」に分けて捉えていました。ハーネスはこの「脳」の側にあたる、AIに何を・どの手順でやらせるかを支える足場です。

AIは、長く動かすほど手順を取りこぼしたり、できたつもりになったりします。だからこそ、その足場をきちんと設計して、安定した仕事をさせる必要がある。OpenAI の[Harness engineering](https://openai.com/index/harness-engineering/)の記事も、まさに「環境を設計し、意図を示し、フィードバックの仕組みを作る」ことが仕事の中心になってきた、と書いていました。

では、その足場を具体的に何で点検すればいいのか。自分は「脳」と「手」の二つで見るようにしています。さきほどの Managed Agents の「脳」と「手」と、ほぼ同じ捉え方です。厳密な分類というより、自分の構成を点検するための地図だと思ってもらえればと思います。

- **脳**：AIに「何を・なぜ・どの手順でやるか」を持たせる部分。LLM そのものに加えて、こちらが用意する知識やルールが入ります。
  - 指標・KPIの定義（売上の計算方法、集計の粒度、対象期間の取り方）
  - 業務・ドメイン知識（自社の用語集、事業の文脈、いつも見ている観点）
  - 分析手順をまとめたスキル（「前年比を出すときは、このデータソースをこの順で集計する」など）
  - 命名規則やレポートの体裁といったルール（AGENTS.md や CLAUDE.md のようなテキスト）
- **手**：与えられたことを実際に実行する手段。ツールやスクリプトです。
  - Tableau MCP（Tableau のコンテンツやメタデータに触る）
  - VizQL Data Service（VDS）や Tableau REST API といった各種 API
  - Tableau Prep / Prep の CLI（データの整形）
  - 集計や検証のためのスクリプト（SQL や Python など）

抽象的な話が続いたので、実際に動かしたものに触れておきます。先日 Tableau Now! で登壇した際に、Tableau MCP と Coding Agent を組み合わせて、分析からレポート生成までをAIに任せる構成を紹介しました。リポジトリの中身は、分析手順を入れた `.agents/skills`、共通ルールを置いた `rules-bank`、目次役の `AGENTS.md`、といった作りになっています。これがまさに、脳と手を設計したハーネスの実体です。

## 「MCPに繋いだだけ」に足りないもの

ここで、冒頭に挙げた「Tableau MCPを使えばAIはデータ分析が出来るの？」に戻ってみます。「ChatGPT に Tableau MCP を繋いだだけ」の状態を、さっきの脳・手・環境に当ててみると、脳はプロンプト程度、手は MCP だけ。なかなかスカスカに見えてこないでしょうか。これで「分析が出来るか」を問うのは、地図のほとんどが空白のまま走り出しているようなものかもしれません。

言い換えると、いま自分のAIに、どんなドメイン知識や業務知識を渡せていて、何が渡せていないのか。それは定型化できるのか。冒頭で書いたこの問いは、まさに脳の部分を点検しています。「AIに何が出来るか」ではなく「自分が何を脳と手に用意できているか」へ。軸足を移すというのは、こういうことなのだと思います。

もう一つの「AIを使ったデータ分析は正しいの？」も、自分は同じ軸足のズレだと思っています。AIに集計や分析を任せて結果が外れたとき、まず疑うべきはモデルの賢さなのだろうか、と。指標の定義、集計の粒度、どの手順で分析してほしいか。そういうものを渡さないまま「正しく分析できるか」を問うのは、AIの能力ではなく、自分の設計を試していることになるのかもしれません。

では、設計が足りているのかどうかは、どう見分ければいいのでしょうか。一つ分かりやすいのは、集計結果を突き合わせてみることだと思います。AIが出した数字がおかしいと感じたら、同じ集計を SQL や VDS（VizQL Data Service）のような決定論的なツールで計算し直して比べてみる。そこでズレるなら、それは計算そのものをAIに任せてしまっている設計の問題です。ツールで出した正しい数字をAIが取り違えるようになって、初めて「能力の限界かもしれない」と疑えばいい。この順番で見ると、「設計を疑え」がただの精神論ではなく、原因を切り分ける手順になってくれます（もちろんモデルにもよりますが、ChatGPT や Claude、Gemini といった今のモデルを使う範囲では、まず設計を疑ってみる価値はあると思っています）。

## 最後に

この記事で一番動かしたかったのは、Tableau MCP の評価のしかたです。Tableau MCP は、結局のところ「Tableau のコンテンツを使うための道具」です。だとすれば、「この道具自体が分析に役立つのか」を試して終わるのは、少しもったいない気がします。問いたいのはむしろ、「この道具を使って、自分たちのデータ利活用にどんな価値を取り込むか。そのために、どんな設計をするか」の方ではないでしょうか。

そう考えると、AIに任せられる範囲が広がるほど、人間が考えるべきことはむしろ「設計」に寄っていくのだと思います。OpenAI の Harness engineering の記事に、「Humans steer. Agents execute.（人間が舵を取り、エージェントが実行する）」という一節がありました。エンジニアの仕事が、コードを書くことから、環境を設計し・意図を示し・フィードバックの仕組みを作ることへ移っていく、と。脳と手をどう組むかを決めるのは、やはり人間の側なんですよね。（ちなみに、「そもそも何を分析するか」「出た示唆を業務の判断にどう変えるか」も人間に残る仕事ですが、これは設計とはまた別の話なので、別の記事で書いてみたいと思います。）

とはいえ、いきなりフルのハーネスを組もうとしなくていいと思っています。今チャットに繋いで分析させている人が明日できる最小の一歩は、たぶん「指標の定義や粒度、用語集をテキスト一枚にまとめて渡す」ことです。それだけでも、指標の取り違えや粒度のズレはかなり減るはずです。これがハーネスの最小単位であり、設計の第一歩だと思います。まずは一枚から、で十分ではないでしょうか。

### 余談：本当は「環境」も三つめにある

正直に言うと、自分はこの「脳」と「手」に、もう一つ「環境」を加えた三つで考えています。環境というのは、AIをどこで・どの権限で動かすか、サンドボックスやパーミッションといった話です。ハーネスを本気で設計するなら、ここも本当は避けて通れません。ただ今回は、概要に留めたかったのと、AI Agentにそこまで触れてこなかったTableauユーザーの方には少し距離のある話かなと思い、あえて脳と手の二つに絞りました。環境の話は、いずれ別の記事で書いてみたいと思っています。

### 余談：MCPは「手」に置いたけれど

MCPは「手」の例として挙げましたが、厳密に言うと、少しだけ「脳」にもまたがっています。MCPが提供するツールには説明文（description）が付いていて、それ自体がLLMに渡る情報だからです。「このツールは何で、どう使うのか」を、AIはその説明から読み取ります。なので説明が雑だと、AIは手の使い方を取り違える。手を用意するときも、その説明＝脳に入る情報の設計からは逃げられない、ということですね。（MCPには Resources のように、もっと直接データや知識を渡す仕組みもありますが、その話はまた別の機会に。）

### 余談：この記事も同じやり方で書きました

正直に白状すると、この記事も似たような進め方で書いています。考えを掘る工程、それを批判的にレビューする工程、最後に自分の判断が要るところだけ自分に戻す工程、というふうに分けて、AIに手伝ってもらいました。ただ、これは「だから良い記事になった」という証拠の話ではありません。あくまで、同じことが皆さんの業務でもできるはず、という地続きの実例として置いておくだけです。

脳と手をどこまで作り込むか、どこが自分のドメインにとっての勘所か、というのは、結局は手を動かしてみないと見えてこない部分だと思います。自分もまだ探っている最中なので、また何か分かったら、その時に書いてみます。

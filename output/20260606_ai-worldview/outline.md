# 記事構成: 「チャットに繋ぐ」から「ハーネスを組む」へ ― Tableau×生成AIで見えている世界を揃える（タイトル仮）

## コアメッセージ
生成AIへの向き合い方の**基準点を置き換えたい**。「生成AI＋Tableau MCPを繋いで正しく分析できるか」ではなく、**ハーネス（脳・手・環境）を設計した上でどう使うか**へ。Tableau×生成AIを語る基準点を、MCP直結デモから「ハーネス込みで考える」に完全に移す。
さらに評価する対象を反転させる: Tableau MCP は“Tableauコンテンツを使うための道具”にすぎない。問うべきは「道具自体が分析に役立つか」ではなく、「**これを使って自分たちのデータ利活用にどんな価値を取り込み、そのためにどんな設計をするか**」。**人間が考えるべきは、その設計だ。**

- **完全情報ゲームは使わない**（批判3／著者判断）。トピック/キーワードを増やすと散らかるため。背骨の説明は「盤面を閉じる＝ドリフトを抑える」に寄せる。
- **時間軸の断定を避ける**（批判1）: 「古い/新しい」でなく「目的が違う」。主語は「AIエージェント領域の標準 vs Tableauコミュニティの現状」。
- **認証前/後の線引きは脚注**（批判2）。
- **結びに最小コストの入口**（批判6）。
- **self-referenceは“橋”であって“証拠”ではない**（批判5/指摘5）。

## 構成

### 導入: 冒頭文（著者執筆・確定）
- 1年間のAI Agent学びを、AI Agent未キャッチアップのTableauユーザーに共有する宣言。技術は概要レベル。
- before/after の問いで軸足を提示:
  - やめたい議論: 「MCP使えばAIは分析できる？」「AI分析は正しい？」
  - してほしい議論: 「どう設計すればいいか？」「いまAIに与えているドメイン/業務知識は何で、何が欠け、定型化できるか？」
- 軸足: 「AI/MCPに何ができるか」→「自分が何を用意・整備すべきか」。これをTableauコミュニティの共通言語に。
- ＝この冒頭が痛点と軸足を提示するので、本文はフレームワーク先行で始めてよい（具体先行の役割は冒頭が担う）。

### 展開1（Ch1）: 脳と手という考え方 ― 今の標準＝広義のハーネス
- 今のAI Agent議論の標準＝広義のハーネスエンジニアリング（足場を整える）。Skills/コンテキスト設計が標準語彙（Anthropic）。
- **広義/狭義のライト補足**: 「広義」と言うのは、作り手が実装する内部ハーネス（狭義）でなく、使い手＝Tableauユーザーが整える外部ハーネスを指すから（watany「ハーネスに関わる混乱と私見」を一言引く）。
- ハーネスとは（福島＝シェフ/調理場、盤面を閉じる、ドリフト）。
- 脳・手（定義2つ）＝足場を点検する地図。Managed Agentsの脳/手分離とほぼ同じ捉え方。※読者属性（AI Agent未キャッチアップのTableauユーザー）を考え「環境（サンドボックス/権限）」は本文から外し、最後の余談で「本当は環境も三つめにあるが今回は割愛」と正直に断る（別記事送り）。
- 具体の錨を早めに: 登壇リポ（.agents/skills, rules-bank, AGENTS.md）＋画像。

### 展開2（Ch2）: 「MCPに繋いだだけ」に足りないもの ― 冒頭2問の回収
- 冒頭の問い①「MCP使えば分析できる？」を回収: 地図に当てるとスカスカ（脳=プロンプト・手=MCPのみ）。
- ドメイン/業務知識の点検＝脳を見ている＝軸足移動の具体。
- 冒頭の問い②「AI分析は正しい？」を回収＝同じ軸足のズレ。楔（精度＝設計）をここに畳む。診断手順＝決定論ツール（SQL/VDS）で再計算して突合→設計起因/能力起因を切り分け。モデル依存の留保も括弧で。

### 結び: 評価する対象を変える ― 人間が考えるのは設計
- 着地（メッセージ本体）: Tableau MCP は“Tableauコンテンツを使うための道具”。だから「この道具自体が分析に役立つか」を評価して終わるのはもったいない。問うべきは「**これを使って自分たちのデータ利活用にどんな価値を取り込み、そのためにどんな設計（ハーネス）を組むか**」。＝導入の「基準点の置き換え」が、ここで「評価対象の置き換え」として閉じる（ブックエンド）。
- 人間の仕事＝設計（軸そのもの、前向きに昇格）: AIに任せられる範囲が広がるほど、**人間が考えるべきは設計**になる。脳・手・環境をどう組み、どんな価値を取り込むかを決めるのは人間。素材＝OpenAI「Harness engineering」の「Humans steer. Agents execute.／環境を設計し・意図を規定し・FBループを作る」を援軍に引く。
- 軸外は別記事に送る（一文だけ）: 「何を分析テーマに選ぶか／示唆を業務判断に変える所」も人に残るが、それは“設計”とは別の話なのでまた書く。深掘りしない。
- 明日できる最小の一歩: いきなりフルのハーネス（Skills/AGENTS.md/Coding Agent）でなくていい。今チャットで分析させている人が**明日踏める最小の一歩＝「指標定義・粒度・用語集をテキスト1枚にして渡す」**。これだけでドリフトが減る＝ハーネスの最小単位＝“設計”の第一歩。まず1枚から。
- self-reference（余談・1段落・軽く）: この記事自体、思考を掘る→批判を当てる→著者の判断だけ戻す、というハーネスで書いた。ただし「だから良い記事」という証拠ではなく、「あなたの業務でも同じことができる」という**地続きの橋**として置く。
- 残す問い: 各軸（脳・手・環境）をどこまで作り込むか、ドメインごとの最適は読者自身の実践に委ねる。

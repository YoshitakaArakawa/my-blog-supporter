# 批判レビュー #01（20260620）

## レビュー対象
VDS は歓迎すべき機能だが、有効化がサーバー単位・default on で「Viewer すらワークブックを迂回してデータソースを直接クエリしうる」統制モデルの変化が AI-readiness にグラデーションのある組織に一斉に及ぶため、サイト単位の有効化トグルを足すべきだ。

初回レビュー。

## 指摘

### 1. アハの「刃」が実機未確認の強い推論に乗っている（記事の根が折れるリスク）
- レンズ: 前提の穴
- 問題: 記事の刃 a（「Viewer の禁止リストに API Access が無い＝Viewer が VDS で直接クエリできる」）は、references.md §4 自身が「単一の明文は無い」「強い推論」「テナントで実機確認1回を推奨」「ここだけ明文が無い」と明記している。コアメッセージ全体がこの一点突破に依存しているのに、その一点が一次ソースの明文ではなく禁止リストの「不在」からの推論。もし実機で「Viewer は API Access を付与してもサイトロール上限で実効にならない」が真なら、記事の刃そのものが折れ、提案（サイト単位トグル）の緊急性の核が崩れる。「禁止リストに無い＝できる」は、明示禁止と明示許可の二値ではなくサイトロール上限という別レイヤーの挙動なので、不在＝可能とは論理的に確定しない。
- 次の一手: 公開前にテナントで Viewer＋API Access の実機クエリ成立を1回検証し、結果を thinking.md / references.md に「確認済み」として固定する。検証できない場合は、記事の構成を「Viewer が直接クエリできる（断定）」から「Viewer がすり抜けうる経路が制度上塞がれていない（要検証の懸念）」へ慎重語に落とす設計を deepen で決める。刃の確度が記事のトーン（警鐘か問題提起か）を規定するので、ここの掘り直しが最優先。

### 2. 「default on」の事実主張に揺れがある（事実誤認なら記事の信頼が傷つく）
- レンズ: 前提の穴
- 問題: thinking.md / brief.md は VDS が `features.VizqlDataServiceDeployWithTsm` = default true で「default on のサーバー機能」だと繰り返す。だが「サーバーで既定有効」と「Viewer が実際にクエリできる」は別の話で、後者には資産ごとの API Access capability の明示付与が要る（references.md §3）。記事が「default on で危ない」と言うとき、読者は「インストールしただけで全 Viewer が全データソースを叩ける」と誤読しうるが、実際は capability ゲートがあるので default では誰も叩けない。著者自身が想定反論1で「API Access は既定 unspecified＝実効拒否」と認めている。つまり「default on」という言葉が、サービスの存在の default と、アクセスの default を混同させる危険がある。
- 次の一手: deepen で「default on」が指すレイヤーを2つに分離して言語化する——(1) サービス自体がサーバー既定で起動している、(2) アクセスは capability 次第で既定拒否。記事では「サービスは既定で立っているが、誤付与・テンプレ・All Users 一括で API Access が許可に化けたときのブラスト半径をサイト単位で局所化できない」という、誤付与リスク＋粒度欠如の論点に重心を移すと、事実とトーンが両立する。

### 3. 想定反論「サイト/プロジェクト既定で API Access 全否定すれば足りる」への返しが弱い
- レンズ: 想定反論
- 問題: thinking.md 観点4-4 と references.md 反論4 で、著者は「今日でも擬似 site deny は作れる」と正直に認めている。これは誠実だが、提案の必要性をかなり削る自爆になりうる。「first-class なトグルが欲しい」の根拠が「opt-out・手作業・継続管理・単一監査スイッチでない」の4点に絞られるが、運用に慣れた管理者は「プロジェクト既定の deny を一度テンプレ化すれば継続管理コストは小さい」「監査も権限レポートで足りる」と返せる。つまり最強反論は「API Access で足りる」ではなく「プロジェクト既定 deny の運用で足りる、トグルは nice-to-have に過ぎない」であり、ここに正面から返せていない。記事が「あれば便利」程度の要望に見えると CTA（担当に声を挙げる）の熱量が出ない。
- 次の一手: deepen で「擬似 site deny では構造的に防げない/コストが跳ねる」具体シナリオを1つ立てる。例: 数百サイト・数千プロジェクトの環境で新規プロジェクトが日々増えるとき、deny の付け忘れが一件でもあれば穴になる（fail-open）。対して site トグルは新規プロジェクトに対しても fail-safe に既定で閉じられる——という「規模が増えるほど擬似 deny が破綻する」論証を入れると、要望が nice-to-have から must に上がる。

### 4. 「普及が止まる＝頓挫」という因果が著者の見立てに留まり、根拠が薄い
- レンズ: 論理の飛躍
- 問題: 書く動機の核は「雑な一斉展開での事故 → 普及が遅れる → Agentic Analytics 構想全体が頓挫」。だが各矢印に飛躍がある。(1) サイトトグルが無いことが本当に「組織まるごと無効」を選ばせるのか（多くの組織はそもそも MCP 導入をまだ検討段階で、無効のまま様子見でも痛くない＝頓挫しない）。(2) 一件の事故が普及全体を止めるという見立ては、他の SaaS 機能でも反例が多い（事故っても機能は残り、改善されて普及する）。この因果が著者の主観的予測のまま提示されると、懐疑的な読者に「煽り」と読まれ、CLAUDE.md の「煽り・断定から価値を作らない」に反する。
- 次の一手: deepen で因果の鎖を「予測」と「観察可能な事実」に分け、断定を引く。「頓挫する」と言い切らず、「全か無かの二者択一しか無いと、準備の整わないドメインを一つでも抱える組織は安全側＝無効に倒れやすい。その分だけ採用が遅れる」までに留めると、検証可能な構造論として立つ。「構想全体の頓挫」まで広げるのは射程過大なので、スコープを「個々の組織の採用ペース」に絞るか、頓挫は仮説とラベルする。

### 5. 「同一 PDS 共用の問い直し」が提案（サイトトグル）と論理的に繋がっていない
- レンズ: 論理の飛躍
- 問題: 観点3-1 の「人間用 PDS をエージェント直クエリ面に流用していいか」は思考として鋭いが、これは PDS の設計判断（専用 PDS を分ける）の話であって、サイト単位トグルの要望とは別レイヤーの解決策に属する。むしろ「専用 PDS を分ければよい」は反論側の武器になりうる（＝トグルが無くても、AI 用に統制済み PDS を別途用意すれば安全に有効化できる、だからトグル不要）。記事に入れると、自分の提案を弱める論点を自分で持ち込む構図になりかねない。
- 次の一手: deepen で「PDS 共用の問い直し」と「サイトトグル」の関係を決める。両立させるなら「PDS 分離は資産単位の正攻法だが全資産を分離し終えるまで時間がかかる。その移行期間を安全に走らせる時間軸の制御がサイトトグル」と、§2と同じ「順序/運用問題」フレームに接続する。接続できないなら、この論点は別記事候補（観点3を触媒どまり）に退避する判断もある。

### 6. CTA が「担当の中の人に声を挙げて」止まりで、読者の行動コストに見合うリターンが描けていない
- レンズ: コアメッセージの弱さ
- 問題: 読後の読者の状態は「自分の現場で辿り直し、担当の Tableau に声を挙げたくなる」。だが「担当者ルートに絞る、Community Ideas は後から」という設計（brief / thinking）だと、読者の行動は「一個人が一営業担当に口頭で要望を伝える」という最も伝播しにくい経路に閉じる。読者から見て、声を挙げても本社に届く保証も他者と束ねる場も無いと、行動の費用対効果が低く、結局「腑に落ちたが動かない」で終わる。CLAUDE.md の北極星「自分の現場で辿り直したくなる」の最後の一押し（実際に動く）が設計で削がれている。
- 次の一手: deepen ではなく主に outline/write の論点だが、動機段階で「読者に何を起こしたいか」を再確認する価値がある。担当者ルートに絞る判断の理由（Community Ideas を今は避ける理由）が読者の行動可能性とトレードオフになっている点を著者に戻し、最低でも「同じ懸念を持つ人がいることを可視化する」場（記事のコメント・SNS での束ね）を CTA に足すか、担当者ルート限定の意図を明示するかを決める。

### 7. 読者リテラシーの非対称が「テーマ」と「執筆制約」を兼ねる構造の自己矛盾
- レンズ: コアメッセージの弱さ / 前提の穴
- 問題: 「最も気づくべき層が、最も気づく装備を持たない」は魅力的な洞察だが、記事の実効性に内在矛盾を作る。a を理解するには権限モデルの深い理解が要る→だが対象読者の多くはそれを持たない→ならば、その読者は記事を読んでも a の重大性を体感できず、CTA（声を挙げる）にも至りにくい。つまり「最も届けたい層に最も届きにくい」記事になる構造リスクがある。これはテーマとして書くこと自体は良いが、記事の到達目標（行動変化）と緊張する。
- 次の一手: deepen で「誰に行動してほしいか」を一段解像度を上げる。a を完全理解できる管理者層（少数だが行動力がある）を一次ターゲットに、一般ユーザー層には「あなたの Viewer 前提が変わる」という体感だけ持ち帰ってもらう、という二段の読者設計を thinking.md に明示すると、非対称が矛盾でなく構成方針になる。

## 総評
最優先は **指摘1（刃 a の実機未確認）** と **指摘2（default on の二義性）**。この2つは事実確度の問題で、ここが揺れると記事全体のトーンと提案の緊急性が根から不安定になる。次点は **指摘3（擬似 site deny で足りる、への返しの弱さ）**——提案の必要性そのものを支える反論なので、「規模が増えるほど擬似 deny が fail-open で破綻する」論証を deepen で立てると要望が nice-to-have から must に上がる。指摘4〜7 はトーンとスコープの調整で、煽り化と射程過大を防ぐために deepen で断定を引く方向に効く。

# テーマブリーフ: VizQL Data Service のサーバー単位有効化への懸念

## テーマ（1行）
VDS は Tableau の Agentic Analytics を支える重要機能として歓迎する。だが「有効化がサーバー単位・default on」で影響範囲が広く、Viewer すらワークブックを迂回してデータソースを直接クエリしうる統制モデルの変化が、準備の整わないドメインにも一斉に及ぶ。サイト単位の有効化制御を足し、組織が安全な速度で採用できるようにすべきだ。

## 読者像（二重）
- 主: Tableau MCP の導入/展開を検討している Tableau ユーザー。管理者・BI/分析プラットフォーム責任者・データガバナンス担当を含むが、**ローコード/ノーコードで入ってきた一般ユーザーも含む**。
- 副: Agentic Analytics・Tableau MCP に関心のある実務者。Tableau の「中の人」（フィードバックを運ぶ先）。
- **リテラシーの非対称（テーマでもあり制約でもある）**: a（Viewer の禁止リストに API Access が無い＝サイトロールの担保が崩れる）を正しく理解するには権限・機能の深い理解が要る。だが対象読者の多くはその装備を持たない。「最も気づくべき層が、最も気づく装備を持たない」——これは記事のテーマの一部であり、同時に「a をかみ砕いて伝える」という執筆上の制約。Tableau の権限モデルを既知の前提として書けない。
- 深さの境界: 一般的な LLM/エージェント論には踏み込まない。c（default on のテクニカル詳細）はアハには使わず提案の補強に留める。

## コアメッセージの方向性（二層）
- **表向き（記事の見た目）**: Tableau MCP の導入/展開を検討している Tableau ユーザーへ、自分の懸念点を共有する。
- **裏メッセージ（真の要望）**: Tableau への制御方法（サイト単位有効化トグル）の追加要望。「機能を止めろ」ではなく「採用の速度を組織が選べる制御を足してほしい」。
- 接地する具体: Viewer の「明示付与しても持てない」禁止リストに **API Access が無い** ＝従来サイトロールが強制していた構造的境界をすり抜ける新種の権限。
- 通底テーマ: 既存の**ワークブック中心の世界観／Viewer ロールと利用可能機能の前提**と、VDS がもたらす**ダイナミックな変化・不整合**。Viewer はその最も鋭い具体例。
- 礼賛と懸念は同一 capability を指す。和解点＝機能の制限ではなく移行速度の制御。

## 興味を引きそうなトピック
1. Viewer の世界観の変化（サイトロールが強制してきた構造的境界 vs API Access のすり抜け）。
2. サーバー単位 default on のブラスト半径と、Tableau 自身の標準パターン（Prep Conductor は site 単位で絞れる）との乖離。
3. 礼賛と懸念の両立＋ CTA（賛同者は担当の Tableau 中の人に懸念を挙げてみてほしい）。

## 既に確定した事実（詳細・出典は references.md）
- VDS 有効化はサーバー一括 TSM（`features.VizqlDataServiceDeployWithTsm`、default true）。site 単位トグル無し。
- VDS は published data source 専用。
- 直接クエリのゲートは API Access capability。Viewer の禁止リストに API Access は無い（＝効きうる）。
- クエリはデータソース層に効く＝workbook の集計・列制御・フィルターは経由しない。

## 未整理・要検討（deepen で掘る候補）
- 書く動機の核: Agentic Analytics 推進派だからこそ「雑な一斉展開での事故」を避けたい当事者性。
- なぜ多くのユーザーがこの統制変化に気づかないのか。
- 「準備できていない」の中身（データソース側／使い手／ワークブック作者の学習／MCP 受容）。
- サイト単位トグル提案への最強反論（「API Access で足りる」）への返し。

# 記事構成: VizQL Data Serviceは歓迎している。だからこそ、サイト単位で有効化させてほしい

> この outline は「話す順の骨」。最終本文は `/write` で散文化する（箇条書きのまま平板に書かない）。
> 方針: 機能の話は章1（アハ）と章3の一部に閉じ込め、章2を「人と組織の準備（主体性）の問い」にして主題を前に出す。

## コアメッセージ
- 表向き: Tableau MCP の導入/展開を検討する人へ、自分の懸念点の共有。
- 裏（真の要望）: Prep Conductor と同型の **サイト単位有効化トグル** を Tableau に足してほしい。
- アハ（山場）: 「Viewer は安全」という、サイトロールが構造的に担保してきた前提が、VDS の API Access で崩れる。
- 主体性の軸: パーミッション制御だけで足りるのか／全ユーザーはついてこれるのか／自社の設計は MCP-ready か。＝「Tableau が悪い」ではなく「我々の準備はどこまでか」。

## 読後の読者の状態
- 「Viewer は読み取り専用だから安全」という前提が崩れることに気づく。
- 自社のデータソース・パーミッション・ワークブック設計が、エージェント直クエリ前提で作られているかを点検したくなる（自分の現場で辿り直す）。
- 煽られて不安になるのではなく「正しく準備して広げたい」という前向きな構えになり、同じ懸念を持つ人と束なって声を挙げたくなる。

## 読者設計（二段・全編で意識）
- 一次ターゲット: アハ a を完全理解できる管理者・プラットフォーム責任者層。
- 二次: ローコード/ノーコードで入った一般ユーザー。「あなたの Viewer 前提が変わる」という体感だけ持ち帰ってもらう。
- 規律: 長さは許容するが「教えるため」に使い「守るため（nitpick 先回り）」には使わない。章2 は問いで開き、チェックリスト化しない。

## 構成

### 導入: 期待から入る ＋ もともと誰のための機能だったか（系譜は圧縮）
展開する思考:
- VDS / Tableau MCP への礼賛。Agentic Analytics の未来に期待。「水を差す批判」でなく「本気で広げたいがゆえの警鐘」と宣言（煽らない）。
- 系譜を2-3文に圧縮: VDS は元々、開発者が特定のユースケースのために Tableau の外からデータソースを使う機能だった。それを MCP/AI Agent が一気に民主化した（素晴らしい）。ただし想定ユーザー像も大きく広がった。
素材:
- ✍️ この懸念に最初に引っかかったときの一次感情（仮置き・著者上書き／社内固有事情に触れない範囲）。
- 系譜の傍証ブログ2本（[use your data your way](https://www.tableau.com/blog/vizql-data-service-use-your-data-your-way) / [beyond visualizations](https://www.tableau.com/blog/vizql-data-service-beyond-visualizations)、著者が内容確認予定）。
接続: 「想定ユーザーが広がった」を軽く立てて、章1「では具体的に何が起きたか」へ。

### 章1: Viewer は本当に「見るだけ」か（世界観の変化＋アハ・山場）
展開する思考（機能の話はアハに絞る）:
- 旧前提: 人間中心の Tableau。Creator/Explorer がワークブックを組み、集計済み・フィルター済みのデータを Viewer に届ける。表示制御はワークブック側（ユーザー関数による出し分け含む）。
  - 具体例: 人事ダッシュボード（集計に使うが非表示の列、所属部署でフィルター）。
- Viewer はサイトロール上限で Download Full Data / Download Data Source を「付与しても持てない」。＝「Viewer にワークブックの窓だけ」がガバナンス実装だった。
- VDS の直接クエリのゲートは API Access。Viewer の禁止リストに API Access は無い。
- アハ: API Access を付与された Viewer はワークブックを経由せず Published Data Source を直接クエリできる。ワークブック側の表示制御は通らない。
  - 精度注（一文）: データソース層に実装した RLS/フィルターは効く。崩れるのは「ワークブック側で表示制御していた」前提。
- 出典（本文で軽く）: `features.VizqlDataServicePermission`「permission check を切ると unlicensed 以外の全 site role が VDS API にアクセスできる」＝制御はサイトロールではなく API Access。
素材:
- 一次根拠: Permissions, Site Roles, and Licenses / Permission Capabilities and Templates / `features.VizqlDataServicePermission`。
- 誠実な短い注（節末）: 「Viewer can use VDS」の単一明示文は未発見。挙動と要件からの帰結。
接続: 系譜（広がった対象）の最も象徴的な具体例。山場。

### 章2: パーミッション制御だけで足りるのか（主体性・readiness の問い）
展開する思考（問いで開く・列挙リストにしない・機能を減らす）:
- 「API Access で制御すればいい」は技術的に正しい。でも立ち止まりたいのはその少し手前。
- パーミッション制御だけで本当に守れるのか: 誤付与（テンプレ一括／All Users のついで）、手作業の運用、規模での付け忘れ（fail-open）。
- 全ユーザーがこの変化についてこれるか: a の理解には権限モデルの深い理解が要るが、Tableau はローコード/ノーコードで広がった。最も気づくべき層が、最も気づく装備を持たない。
- 自社の設計は MCP-ready か: データソース設計・パーミッション設計・ワークブック設計は、エージェントが直接叩く前提で作られているか。人間がワークブック越しに見る前提で整えた PDS を、そのままエージェントの問い合わせ先にしていいか（問いとして残す）。
- 準備度は組織内で不揃い＝グラデーション。
- 主体性: 「Tableau が悪い」ではなく「我々の準備はどこまでか」。動機（ちゃんと広げたい）に直結。
素材:
- thinking 動機・観点3。RLS 実装位置の話は深追いせず触媒どまり。
接続: 「準備にグラデーションがある」を立て、章3「なのに有効化の制御は両極しかない」へ。

### 章3: サイト単位で有効化させてほしい（要望・粒度は圧縮）
展開する思考:
- 準備にグラデーションがあるのに、有効化の制御は**両極しかない**——全体（サーバー）か個別（資産＝パーミッション）か。組織・ドメイン＝サイト単位の中間が無い。
- 機能の具体は圧縮（前面に出さない・リンク中心）: Server はサーバー単位で OFF にはできるがサイト粒度なし。Cloud はサービス自体を止める手段が無く制御はパーミッションのみ（既定 deny）。＝粗いスイッチだけ／細かいスイッチだけ。
- 既に Tableau 自身がやっているパターン: Prep Conductor は「サーバーで有効化 → サイト単位で絞れる」。VDS だけ無い。
- なぜ一定の解か: 準備できたドメインから順に広げられる／新規にも fail-safe に閉じる／統制済み PDS への移行に時間の余裕。
素材:
- 圧縮詳細のリンク: cli configuration set / VDS server process / Prep Conductor サイト設定。「Cloud は野放し」と書かない（既定 deny）。
接続: 章2の「準備のグラデーション」への、具体で最小の解。

### 最後に: 採用のペースを、組織が選べるように（CTA）
展開する思考:
- 着地は採用ペース（「頓挫」は使わない）。全か無かしか無いと、準備の整わないドメインを1つでも抱える組織は安全側＝無効に倒れやすく、その分採用が遅れる。
- CTA（A）: 担当の Tableau のなかのひとに声を挙げてみてほしい＋記事コメント/SNS で可視化。数が見えると個人の困りごとが共有された声になり、なかのひとの後ろ盾になる。
残す問い:
- あなたの組織のデータソースとユーザーは、いまグラデーションのどこに立っているか。

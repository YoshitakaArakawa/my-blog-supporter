# 思考の深掘り: VizQL Data Service のサーバー単位有効化と Tableau の世界観の変化

## 今回のスコープ（手順2で確定）
- **本筋（1文）**: VDS は Agentic Analytics を支える歓迎すべき機能だが、有効化がサーバー単位・default on で、「Viewer すらワークブックを迂回してデータソースを直接クエリしうる」という統制モデルの変化が、AI-readiness にグラデーションのある組織へ一斉に及ぶ。サイト単位の有効化制御を足し、組織が安全な速度で採用できるようにすべきだ。
- **深掘りリスト（4点）**:
  1. 書く動機の核
  2. 世界観の不整合と"気づかれなさ"（a がアハ／リテラシーの非対称）
  3. 「準備できていない」＝ AI-readiness のグラデーション（概念2点に格上げ）
  4. サイト単位トグル提案の妥当性（反論への返し）

## コアメッセージ（磨きこみ後）— 二層
- **表向き**: Tableau MCP の導入/展開を検討している Tableau ユーザーへ、自分の懸念点を共有する。
- **裏**: Tableau への「サイト単位有効化トグル」追加要望。「機能を止めろ」ではなく「採用の速度を組織が選べる制御を足してほしい」。
- **アハ（記事の刃）**: Viewer の「明示付与しても持てない」禁止リストに **API Access が無い**。これまでサイトロールが構造的に担保してきた「Viewer はデータソースに直接届かない」前提が崩れる。
- **和解点**: 礼賛も懸念も同一 capability を指す。求めるのは機能の制限ではなく、移行速度の制御（グラデーションを段階展開に変換する手段）。

## 読後の読者の状態
- 「Viewer は読み取り専用だから安全」「ユーザーが見るのはワークブックが見せたものだけ」という前提が、VDS で崩れることに**気づく**。
- 自社の AI-readiness に**グラデーション**があると認識し、サーバー一括の二者択一の危うさを腑に落とす。
- 「サイト単位制御」という要望に共感し、自分の現場で辿り直し、担当の Tableau に声を挙げたくなる。
- 煽られて不安になるのではなく、「正しく準備して広げたい」という前向きな構えになる。

## 書く動機（深掘り後）
- [著者回答] 機能も世界観も広げたい立場。同時に長く Tableau を使ってきた当事者として現実を見ている。中心の懸念は「事故そのもの」より**「普及が止まること」**。
- [著者回答] Tableau はもともと**人間中心**のデータ利活用の世界観。VDS も 2025.1 リリースで、Agentic な世界を前提に設計された機能ではない。だがこの1年で Agent は急進化し Tableau MCP も出た。世界は人間中心→Agentic へ動いている——**ただし一様ではない**。データもユーザーも AI-ready 度にグラデーションがある。
- [著者回答] サイト単位の安全弁が無いと、採用可否がサーバー全体の二者択一になる。準備の整わないドメイン（人事・機密系）を一つでも抱える組織は、安全側に倒して「組織まるごと無効」を選びやすい。**グラデーションを段階展開に変換する手段の欠如が、採用を全か無かに変え、無に倒れやすくする**＝頓挫の経路。これが放置されれば Agentic Analytics Platform 構想は頓挫し、Tableau MCP 展開の壁になる。
- [著者回答] だから自分のブログで一石を投じ、読者の Tableau ユーザーを巻き込み、いずれ皆が気づく問題を前倒しで可視化したい。

## 各観点の深掘り

### 観点1: 書く動機の核（推進派が、あえて懸念を書く）
- 主張: 推進したい立場の人間が公に懸念を書くのは「水を差す」リスクを伴うが、それでも書く。理由は、雑な一斉展開での事故が普及を遅らせ、構想全体を頓挫させると見ているから。懸念は防御ではなく**普及を本気で望むがゆえの前倒しの警鐘**。
- 根拠・背景: [著者回答] 人間中心設計の Tableau × Agentic の急進化、という時間軸のズレ。VDS は Agentic を念頭に作られていない。
- 素材: 「礼賛と懸念は同じ機能を指す」「全か無か → 無に倒れやすい」。

### 観点2: 世界観の不整合と"気づかれなさ"
- 主張: 記事の刃は **a**（Viewer の禁止リストに API Access が無い＝サイトロールの担保が崩れる）。**b**（VDS のクエリはワークブックを経由せずデータソース層に効く）は機能性質上あたりまえで、a の技術的裏付けとして添える。**c**（default on のサーバー機能）はテクニカルすぎるのでアハには使わず、提案（影響範囲＝なぜサイト制御が要るか）の補強に留める。
- "気づかれなさ"の構造的原因: [著者回答] a を正しく理解するには権限・機能の深い理解が要る。だが Tableau はローコード/ノーコードで一般ユーザーを巻き込んで普及してきた。**最も気づくべき層が、最も気づく装備を持たない**という非対称。これはテーマの一部であり、同時に「a をかみ砕いて伝える」という執筆上の制約でもある（権限モデルを既知の前提にできない）。
- 根拠（一次ソース）: Permissions, Site Roles, and Licenses（Viewer は Download Full Data / Download Data Source を明示付与されても持てないが、その禁止リストに API Access は無い）。詳細は references.md。
- 素材: 「Viewer は安全」という素朴な安心 → a で崩れる、という一点突破の構成。

### 観点3: 「準備できていない」＝ AI-readiness のグラデーション（概念深掘り）
- 主張: 「準備」は技術チェックリストの話に矮小化しない。概念として効くのは2点。
  1. [著者回答] **AI 用とワークブック用で同じ PDS を共用していいのか**——人間中心設計で作った PDS を、そのままエージェントの直クエリ面に流用していいか、という**利用モデルの問い直し**。AI 向けに統制済みの専用 PDS を分ける設計判断もあり得る。
  2. [著者回答] **部門でルールが違う＝グラデーション**——部門は多くの場合サイト/プロジェクト境界に対応し、各々ガバナンスルールが異なる。サーバー単位の二者択一はこの部門差を構造的に表現できない。→ 点4（サイト単位）の論拠の核。
- 補足（触媒として軽く・列挙ルーブリックにはしない）: RLS など表示データ制御がワークブック側実装だと VDS 直クエリで効かない（データソース側実装なら効く）。利用者が正しい AI 利用を理解しているか。
- 書き方の規律: [著者回答] 分量は長くてよいが、長さは**「教えるため」に使い「守るため」には使わない**。技術記述の採否テスト＝「読者が変化を理解するのに要るか／著者を細かい反論から守るためか」。後者は脚注か削る。

### 観点4: サイト単位トグル提案の妥当性（反論への返し）
- 主張: 求めるのは Prep Conductor と同型の**サイト単位有効化トグル**（サーバーで有効化 → サイト単位で絞れる、という Tableau 自身の標準パターン）。VDS だけこの site 粒度が無い。
- 最強反論への返し（references.md「想定反論への備え」と同じ）:
  1. 粒度の種類が違う。API Access は資産ごとの fine-grained **allow**。提案は site/ドメイン単位の粗い **deny（キルスイッチ）**。defense in depth は両方を要する。
  2. 負担の反転/ブラスト半径。サーバー一括 default on は「全 data source を準備し終えるまで安全に有効化できない」を強制。サイトトグルは準備済みから順にペースを区切れる。
  3. 誤付与の同時多発（テンプレ/All Users/一括操作で API Access が許可に化ける）をサーバー有効化が全サイト同時に開きうる。site バックストップで局所化。
  4. 正直に認める: 今日でも「サイト/プロジェクト既定で API Access 全否定」で擬似 site deny は作れる。ただし opt-out・手作業・継続管理・単一監査スイッチでない。だから first-class なサイトトグルを求めるのは妥当。
- 根拠（一次ソース）: Prep Conductor のサイト単位設定（references.md）。
- [AI回答] 制御粒度の精密化（Server/Cloud）: Server は TSM でサーバー単位の無効化はできるがサイト粒度が無い。Cloud は TSM 無しでサービス自体の無効化手段が無く、制御は資産ごとの API Access のみ（既定 deny）。→ Server＝粗いスイッチのみ／Cloud＝細かいスイッチのみ／両者とも「サイト単位」という中間粒度が無い。サイトトグルは両方の穴を埋める。「Cloud は野放し」とは書かない（既定 deny）。

### 観点5: VDS の系譜（前提の変化）
- [著者回答] 主張: VDS は MCP/Agent 時代の前から開発されていた機能。当初の狙いは「開発者が特定のユースケースのために、Tableau の外からでも Tableau のデータソースを使えるようにする」こと。可視化を介さずデータを取り出す API。
- [著者回答] 紹介ブログは技術者向け（観察）。元々「リスクを正しく理解し、正しい設定の上で使う人たち＝開発者」を想定して作られた（intent は推測ラベル）。そこへ MCP/AI Agent が来て民主化した（素晴らしい）。だが想定ユーザーが、リスクを正しく認知しない層・ユースケースにも広がった。Tableau の失敗ではなく「対象の変化」。
- 素材: ブログ2本（[use your data your way](https://www.tableau.com/blog/vizql-data-service-use-your-data-your-way) / [beyond visualizations](https://www.tableau.com/blog/vizql-data-service-beyond-visualizations)、著者が内容確認予定）。
- 役割: 記事の章1。Viewer aha（章2）への布石＝「広がった対象の最も象徴的な具体例が Viewer」。

最初に立場を書いておくと、自分はVizQL Data Service（VDS）もTableau MCPも歓迎している側です。可視化を介さずにデータソースへ直接問い合わせられるこの仕組みは、TableauがAgentic Analyticsへ踏み出すための土台だと思っています。だからこの記事は「VDSは危ない」という警告ではありません。むしろ逆で、本気で広がってほしいからこそ書いています。

正直に書くと、自分がいちばん気にしているのは事故そのものより、「普及が止まること」の方です。雑な一斉展開でどこかが事故を起こすと、その一件で組織全体が安全側に倒れ、せっかくのAgentic Analyticsの採用が遅れてしまう。推進したい立場の人間が公に懸念を書くのは少し水を差すようで気が引けるのですが、いずれ皆が気づくであろう論点を、前倒しで一度言葉にしておきたいと思いました。

その懸念を一言にすると、VDSの有効化が「サーバー単位」でしか制御できないことです。エージェントにデータを直接触らせる準備は、部門やドメインによってばらつきがあります。なのに有効化は全体でオンにするか、資産ごとにパーミッションで絞るかの両極しかなく、その中間がありません。準備の整わないドメインを1つでも抱えていると、組織はまるごとオフに倒れがちになります。だからこの記事では、Prep Conductorのように「サーバーで有効化して、サイト単位で絞れる」制御を足してほしい、という要望を共有したいです。

## VizQL Data ServiceとTableau MCPは人間中心のTableauの世界観を変えた

Tableauは長い間「人間中心」のデータ利活用ツールでした。
CreatorやExplorerがワークブックでダッシュボードや可視化を組み、集計済み・フィルター済みのデータを人（多くはViewer）に届ける世界観でした。
どの列を見せるか、どうフィルターするかは、基本的にワークブック側で制御されていました。ここでのフィルターはディメンションやメジャーでのフィルターも、閲覧者の属性によって変化する、ユーザー関数を用いた制御も含みます。
（例：人事情報ダッシュボード - 集計には使用するが表示しない列がある、表示データは閲覧者の所属部署にフィルターされる、など）

一方で、2025.1にてVizQL Data Service（VDS）という機能がリリースされました。端的に言えば「ダッシュボードを介さずに、その裏のデータソースへ直接問い合わせる」ための機能です。

これまでTableauは、ワークブックを通して、整えられたデータを人に届けてきました。VDSはその窓を経由せず、ワークブックの裏側にあるPublished Data Sourceに対して、プログラムやAI Agent（Tableau MCP）から直接クエリを投げ、欲しい列・集計・フィルターを指定してデータを取り出します。

ここで重要なポイントがあります。VDSはViewerユーザーも利用可能です。
例えばTableau Server設定値のページには、以下の記述があります。
'''データ ソースのパーミッション ルール「API アクセス」の権限チェックを有効にします。...(中略)...すべてのサイト ロール (ライセンスなしのロールを除く) が VizQL データ サービス API にアクセスできるようになります。'''

厳密に言えば、データソースにVDSを使用してクエリをするためには、実用上は「APIアクセス」パーミッションを、そのデータソースに対して設定する必要があります。上記の記述がある「features.VizqlDataServicePermission」は、このパーミッション設定に関する項目です。

ただしパーミッション設定で制御するにしても、サイトロールでの制御ではないため、パーミッション付与されているViewerはデータソースにクエリが出来るようになります。

つまり、今までの「ワークブックで表示制御をすれば良い」という世界観は、VDSが有効化されている環境では、2025.1以降では崩れていました。

設定値とパーミッション設定によっては、ワークブック側のフィルターや表示項目に関わらず、Viewerユーザーも集計前のデータが確認できます。

これはVDSが紹介され始めた頃の公式製品ブログからの推測ですが、Tableauの各種APIがそうであったように、VDSはあくまでも技術者向けの機能として開発されていたと思います。ViewerがPATを発行し、個人的な目的のために直接VDS APIを使用してデータを取得しに行くようなユースケースは、この機能の開発時点では想定していたのでしょうか...?
VizQL Data Service from Tableau: Use Your Data, Your Way (2024/8/8)
VizQL Data Service: Extend Your Data Beyond Visualizations (2025/3/19)

また2025.1リリースされた2025年2月時点では、AI Agentの話題は現在ほど本格化していないと記憶しています。MCPの議論が本格化し始めたのは2025年の初頭ごろのようです。ちなみにTableau MCPのリリースは2025年6月ですね。

これは自分の認識としては、VDSは開発者ユーザーがパーミッション設定などを十分意識した上で利用する分には、まったく問題なかったと思います。
2024年～2025年はHeadless BIの議論も熱かったように記憶しており、開発者やエンジニアを念頭に、Tableauデータソースをそのまま使える機能が開発されていたことは、むしろ素晴らしいことだったと思います。

一方で、現在はどうでしょうか？
Tableau MCPはVDSを民主化しました。ViewerユーザーはAIとTableau MCPを使って、VDSを好きなように利用できます。

## なぜ「Viewerは安全」だと思えていたのか

ここで少し立ち止まって、これまで自分たちが「Viewerは見るだけだから安全」と感じていた、その安心の正体を考えてみたいです。

Tableauの権限モデルでは、サイトロールがそのユーザーに持てる権限の上限を決めています。Viewerには、データソースに対する「Download Full Data」や「Download Data Source」を、たとえ明示的に付与しても持たせられません。公式にもこうあります。

> "a user with a site role of Viewer will never be able to download a data source even if that capability is explicitly granted to them on a specific data source."

つまり、Viewerが生データに手を伸ばせなかったのは、運用者がいつも気をつけていたからではなく、サイトロールが構造的に塞いでいたからでした。明示的に許可しようとしても効かない、という強い保証だったんですね。「Viewerにはワークブックという窓だけ」という運用は、この保証の上に乗っていました。

ところが、この「明示付与しても持てない」禁止リストに、「API Access」は入っていません。

公式の権限ドキュメントを並べて読むと分かります。
- [Permissions, Site Roles, and Licenses](https://help.tableau.com/current/server/en-us/permission_license_siterole.htm)
- [Permission Capabilities and Templates](https://help.tableau.com/current/server/en-us/permissions_capabilities.htm)

API AccessはData SourcesのPublishテンプレートに属するcapabilityで、Viewerが普通に保持できる「View」「Connect」と並んで、付与すれば実効になります。VDSでデータソースをフルにクエリするのに必要なのはこの3つ（View、Connect、API Access）で、Viewerはどれもサイトロールの上限に引っかからず持てます。先ほどの「権限チェックを切ると、ライセンスなしを除く全サイトロールがVDS APIにアクセスできる」という記述とも整合します。Viewerはサイトロール層では塞がれておらず、唯一のゲートはAPI Accessだけ、ということです。

だから、API Accessを付与されたViewerは、ワークブックを経由せず、Published Data Sourceを直接クエリできます。サイトロールが構造的に担保してきた境界を、API Accessという新種の権限がすり抜ける。今回いちばん伝えたかったのは、この一点です。

誤解されたくないので一文だけ補足すると、崩れるのは「ワークブック側で表示を制御していた」前提の方です。データソース側に実装したRLSやデータソースフィルターは、VDS経由でも効きます。だから後で触れる「制御をデータソース側に移せばよい」という指摘は正しい。崩れたのは、データソースの手前にあったワークブックという窓の方なんですね。

ちなみに、「ViewerはVDSを使える」と一文で言い切った公式記述を、自分はまだ見つけられていません。ここは権限チェックの挙動とcapabilityの要件からの帰結として書いています（断定はしますが、出典の性質は正直に開示しておきます）。

## パーミッション制御だけで足りるのか

ここまで読んで「だったらAPI Accessをちゃんと管理すればいい」と思った方は、正しいです。技術的にはその通りで、自分も反論できません。ただ、立ち止まりたいのはその少し手前です。パーミッション制御「だけ」で、本当に守りきれるのか。

例えば、テンプレートの一括適用や「All Users」へのついでの付与で、API Accessが意図せず許可に化けることがあります。資産ごとのきめ細かい許可は、付け忘れや付けすぎも規模に比例して増えます。数百サイト、数千プロジェクト、新規プロジェクトが日々増えていく環境で、すべてを手作業で正しく保ち続けられるでしょうか。1件の付け忘れが、そのまま穴になる（fail-open）。

もう1つ気になるのは、この変化に全員がついてこれるか、です。さっきの話（Viewerの禁止リストにAPI Accessが無い）を正しく理解するには、Tableauの権限モデルと機能の、それなりに深い理解が要ります。でもTableauは、ローコードやノーコードで多くの人を巻き込みながら広がってきました。いちばん気づくべき層が、いちばん気づく装備を持っていない。この非対称は、煽りたいわけではなくて、民主化がうまくいったことの裏側だと思っています。

そして、たぶんこれがいちばん大きい問いです。自社のデータソース設計、パーミッション設計、ワークブック設計は、エージェントが直接叩く前提で作られているでしょうか。人がワークブックという窓越しに見ることを前提に整えてきたPublished Data Sourceを、そのままエージェントの問い合わせ先にしてよいのか。AI向けには統制済みの専用データソースを分ける、という設計判断もありえます。これは答えを押し付けたい話ではなくて、一度立ち止まって問い直す価値がある、というところまでです。

こうして見ると、「準備ができているか」は組織の中で一様ではないことに気づきます。部門ごとにルールが違い、機密の度合いも違う。準備度にはグラデーションがあります。これは「Tableauが悪い」という話ではなくて、「我々の準備はどこまでできているか」という、こちら側の問いなんですね。

## サイト単位で有効化させてほしい

準備にグラデーションがある。なのに、有効化の制御は両極しかありません。サーバー全体でオンにするか、資産ごとにパーミッションで絞るか。組織やドメインの単位、つまりサイト単位という中間が無いんです。

機能の細かい話は本筋ではないので圧縮します。Tableau Serverは、サーバー単位でならVDSをオフにできます（TSMで`vizqldataservice`プロセスを止める）。でもサイト単位の絞りはありません。Tableau Cloudはそもそもサービス自体を止める手段が無く、制御は資産ごとのAPI Accessだけです。API Accessは既定では付与されておらず実効的に拒否なので、「Cloudは野放し」ではありません。要は、粗いスイッチしか無いか、細かいスイッチしか無いか、のどちらかなんですね。

- [Tableau Server VizQL Data Service（無効化の手順）](https://help.tableau.com/current/server/en-us/server_process_vizql-data-service.htm)

ここで思い出したいのが、Tableau自身がすでに同じ形を持っていることです。Tableau Prep Conductorは「サーバーで有効化して、サイト単位で絞れる」。公式にもこうあります。

> "If you have multiple sites, you can selectively turn off Tableau Prep for Server for individual sites."

- [Tableau Prep Conductor のサイト設定](https://help.tableau.com/current/server/en-us/prep_conductor_configure_server_site_settings.htm)

VDSにだけ、このサイト単位が無い。Web Authoringにも同じサイト単位のトグルがあることを思うと、VDSがTableau自身の標準パターンから外れているように見えます。

サイト単位のトグルがあれば何が嬉しいか。準備できたドメインから順番に広げられます。新しく作られたサイトやプロジェクトにも、既定で安全側に閉じておける（fail-safe）。そして「統制済みのデータソースに移行する」のが正攻法だとしても、その移行を走らせている間、安全に時間を稼げます。

正直に書くと、今日でも「サイトやプロジェクトの既定でAPI Accessを全部拒否する」運用で、擬似的にサイト単位のdenyは作れます。ただしそれはopt-outで、手作業で、ずっと管理し続ける必要があって、1つの監査スイッチにはならない。だからPrep Conductorと同じ、first-classなサイトトグルが欲しい、という要望です。

### 余談：Tableau MCPの「サイト設定」はこの代わりになるか

「Tableau MCPにはサイト単位の設定があるじゃないか」と思った方もいるかもしれません。実際、Tableau MCPには、サイト管理者がREST API経由でサイト単位に設定を上書きできる仕組みがあります（Tableau 2026.2以上）。出すツールの絞り込み（`INCLUDE_TOOLS`）や結果件数の上限（`MAX_RESULT_LIMIT`）といった項目を、サイトごとに差し替えられます。

- [Tableau MCP / Site Settings](https://tableau.github.io/tableau-mcp/docs/configuration/mcp-config/site-settings)

ただ、これは今回の話の代わりにはならないと自分は思っています。理由は2つあります。

1つは、ここで制御できるのは「MCPというツール層」の挙動であって、VDSそのものへのアクセス境界（API Access）ではないこと。出すツールや結果の上限は絞れても、PATとAPI Accessさえ持っていれば、MCPを介さずVDSを直接叩く経路はそのまま残ります。エージェント任せでない素のAPI利用は、この設定の射程の外なんですね。

もう1つは、その設定を守るかどうかが、MCPプロセス側の環境変数（`ENABLE_MCP_SITE_SETTINGS`、既定はオン）次第だということ。これをオフにすると、ドキュメントにある通り、サーバーはサイト設定の上書きを「取得も適用もしない」。そしてローカルにstdioでMCPを立てる構成では、その環境変数を握るのはユーザー本人です。つまり実効的に効くのは、組織がMCPのランタイムを握っている（マネージドやサーバー集中の）構成のときくらいで、ユーザーが自分の手元でサーバーを起動できるなら、管理者のサイト設定はオプトアウトされ得ます。

むしろ面白いのは、MCPという一段上の層にはサイト単位の設定があるのに、その下のVDS自体にはサイト単位の制御が無い、という逆転です。欲しいのは、ツール層ではなくデータ層に効くサイトトグルの方なんだと思います。

## 最後に

この記事で言いたかったのは「VDSは危ない」ではありません。むしろ逆で、VDSとTableau MCPが開くAgentic Analyticsの世界を、本気で広げたいからこそ、という話です。だからこそ、雑な一斉展開で事故が起きて、せっかくの普及が遅れてしまうのを避けたい。

全か無かの二者択一しか無いと、準備の整わないドメインを1つでも抱える組織は、安全側、つまり「組織まるごと無効」に倒れやすくなります。その分だけ、採用が遅れます。サイト単位のトグルは、このグラデーションを段階的な展開に変えるための、いちばん小さな道具だと思っています。

もしこの懸念に共感してもらえたら、担当のTableauの中の人に、ぜひ一度この話を挙げてみてください。あわせて、この記事のコメントやSNSで「自分も気になっている」と可視化してもらえると嬉しいです。一人の困りごとは流れていきますが、同じ声が集まると、中の人が社内で動くときの後ろ盾になります。

最後に1つ、問いを残します。あなたの組織のデータソースとユーザーは、いま、このグラデーションのどのあたりに立っているでしょうか。

自分もまだ確信を持てていない部分があります。権限チェックを司る設定の正式な名前の最終確認や、Tableau Cloud側の挙動の原文確認は、まだ宿題として残っています。新しい事例や、はっきりした事実が出てきたら、その時にまた書いてみます。

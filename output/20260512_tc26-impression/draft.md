この記事は、Tableau Conference 2026 (TC26) の Main Keynote と、現地で浴びた各種セッションを踏まえて、Tableauユーザーの一人である自分が今、何を考えて何を感じているかを書いてみる、思索メモのようなものです。

機能の紹介や発表の要約は基本的にしないつもりです。一次情報は冒頭のリンクから辿ってもらえればと思います。あくまで、Keynoteで提示された「Architect」というキーワードを自分なりに受け取った時に、Tableauユーザーとして、データに関わる人として、これから何を考えてどこに生息地を広げていくべきか、という個人的な思索です。

なお、Tableau Next や Agentforce はまだ実務で使ったことがないので、想像と見聞きの範囲が混じります。その辺りは多めに見てもらえると助かります。

---

## TC26 Keynote で提示された「Architect」というキーワード（簡単に）

今回のKeynoteで自分が一番強く受け取ったメッセージは、Tableauが「agentic analytics platform」として再定義され、これからのTableauユーザーは:

- **Knowledge Architect**（データやセマンティクスを設計する人）
- **Decisions Architect**（意思決定の流れを設計する人）
- **Agentic Architect**（AIエージェントを設計する人）

という、3つの「Architect」のいずれか、もしくは複数の組み合わせとして動いていきましょう、というものでした。

<!-- 画像: Keynoteの「3つのArchitect」スライド -->

「Architect」というキーワード自体には、自分は完全に同意します。「データを起点に意思決定とアクションをデザインする人」への役割拡張という言い方は妥当だと思いますし、ここ数年、自分も似たようなことを言ってきたつもりです。

逆に、もし今回のKeynoteの軸が、相変わらず「ダッシュボードの作り方」「可視化の機能」みたいな話に閉じていたら、自分は正直、少しがっかりしていたかもしれません。「Help people see and understand data」のフレーズが薄れ、もっと広い世界に開いていく方向性を強く打ち出してくれたことは、歓迎しています。

ここまでは、ほぼ祝福の話です。ここから少し、踏み込んだ話をしてみたいと思います。

## 一方、我々にロールの広がりを受け入れ実践する準備は、どこまであるか

Tableau は「Architect」という言葉を用いて、データに関わる我々のロールの広がりを提示してくれました。これ自体には、前段で書いた通り、自分は完全に同意しています。

ただ、ここで一度立ち止まって考えてみたいのは、我々のほうに、そのロールの広がりを受け入れて実践していく準備が、現在どこまでできているか、ということです。

Tableauユーザーである自分が言うのも変かもしれませんが、Tableau (Cloud / Server / Desktop。以降「Tableau Classic」と呼ばせてください) の製品世界観の中だけにインプットや知識を閉じたままで「Architect」として動こうとすると、Architect の最適解はなかなか見えてこないように思います。

そもそも Tableau の良さの一つは、可視化機能それ自体よりも、他のデータプラットフォームや業務システムとの「接続性」「連携力」にあると自分は思っています。Salesforce、Snowflake、Databricks、各種 SQL データベース、外部認証基盤、Slack 連携、REST API 経由の自動化など、Tableau が真価を発揮する場面では、ほぼ必ずと言ってよいほど周辺製品との接続が前提になっているはずです。

「Architect」として動こうとした時、その「Tableau の良さ」をそのまま延長すれば、関連する製品 — Tableau Next、Agentforce、Data 360、そして今年4月に発表された Salesforce Headless 360、さらに MCP 経由で接続されうる他社のエージェント基盤 — に、自分たちの視野や知識を広げていく必要が出てくるように思います。逆に言えば、そこに踏み込まないと、せっかくの「Tableau の連携の良さ」を Architect の設計に生かしきれない、と感じています。

この感覚は、TC26 のいくつかのセッションを横断して受け取ったときに、立体化されて見えてきました。

- **Will の Tableau Keynote** が Architect への変化を抽象的に提示し
- **Tristan Handy のセッション** が「変化を embrace するマインドセット」と業界全体の文脈を与えてくれて
- **Adrian の "5 Things Tableau Developers Should Know About Tableau Next"** が、Tableau Developer 目線で「既存の前提を Tableau Next ではアップデートしないと通用しない」ことを具体的に示し
- **"10x Your Tableau Workflow with AI and Claude Code"** が、Claude Code から Tableau の API や embedding を触る実践を見せてくれた

それぞれは独立したセッションでしたが、横断して受け取ると「Architect になっていくには、Tableau Classic の世界観だけに閉じていては届かないところまで、自分たちの視野を広げる必要がある」という感覚が、確信に近づいていきました。

つまり、ロールの広がりを受け入れる準備とは、Tableau Classic という製品に何かが足りないという話ではなく、ユーザーである我々の側のインプット・知識・視野を、Tableau の周辺、その先にあるエコシステムまで広げていくかどうか、という話なのだと自分は思っています。

### 余談：Tableau / Salesforce 側も、本当はもっと話したかったように見えた

ここで一つ、共感的な観察を差し込ませてください。

Keynote や横断的なメッセージングでは、Tableau Next / Agentforce / Headless 360 のアーキテクチャや関連製品との接続を深く掘る場面が、自分の期待よりは少なかった、というのが正直なところです。

ただ、これはおそらく「Tableau / Salesforce 側に語る材料がない」という話ではなくて、TC という場の性格 — 既存の Tableau Classic ユーザーへの配慮、Tableau ブランドの立ち位置 — に合わせて、意識的に抑制していたのではないか、というふうに自分には見えました。

その証拠と言えそうなのが、個別の Tableau Next 関連セッション — たとえば "5 Things Tableau Developers Should Know About Tableau Next"、"The End of Busywork: Agentic Prep, Modeling, and Viz"、"Custom AI Architectures: Fueling Agents with Tableau" などでは、Salesforce エコシステムや関連アーキテクチャの話も自然に展開されていたことです。

つまり、Keynote / 横断メッセージングと、個別の Next セッションとの間に、明確な温度差があった。これは批判ではなく、「Tableau / Salesforce 側の言いにくさ」への共感的な観察として書いています。

そして、だからこそ、その温度差を読み取れる側のユーザーが、自分から境界を引き直していく必要があるのかもしれない、と感じました。

## AI / Agent 側でも、視野を広げないと「Architect」にはならない

視野を広げる必要があるのは、Tableau の周辺エコシステム側だけではないように感じています。AI / Agent 側でも、同じことが言えるのではないかと思っています。

「Architect」を本気でやるなら、ChatGPT や Claude をプロンプトでうまく使いこなす、というレベルの話だけでは足りなくなってくる気がしています。少なくとも以下のような、もう少し設計寄りの語彙を持って動く必要があるように思います。

- **「Skills」**（Claude Skillsのような、AIに任せる作業の「型」をパッケージとして定義する考え方）
- **「ハーネスエンジニアリング」**（狭義/広義で、LLM を取り巻くコード — どの情報を保持し、取得し、提示するかを設計する仕事）
- **「コンテキストエンジニアリング」**（どの文脈をどう注入するかを設計する仕事）
- **「Managed Agents」**（自律的に動くエージェントの設計と運用）

このあたりを軽視してしまうと、結局「ChatGPTに聞いたらこう答えました」みたいな話の延長で止まってしまって、本当の意味で「Architect」する側には立てない、と感じています。

この観点を後押ししてくれる読み物として、Tristan Handy の「[Five things I believe about the future of analytics](https://roundup.getdbt.com/p/five-things-i-believe-about-the-future)」を強くおすすめしたいです。

特に **Thing 5: Harnesses are a leverage point** は、Stanfordの論文『Meta-Harness』を引いて「ハーネスを変えるだけで6倍の性能差が出る」「ドメイン特化したハーネスが汎用ハーネスを大きく上回る」という話をしています。これはまさに、「Architect」がやるべき設計仕事の中核がどこにあるか、を示している話だと自分は読んでいます。

加えて、**Thing 1**（アナリストがテクニカル化している）や **Thing 4**（エージェント発起のクエリが、人間発起のクエリを上回る）も、「アナリストがテック側、上流側に押し上げられている」という流れの直接的な根拠になっています。

つまり、「Architect」として動くなら、Tableau や BI、可視化の手前にあるこういう領域にも、踏み込んで知っておく必要がありそうだ、ということです。

## これは Tableau だけの話ではなく、業界全体の流れの一断面

ここまで Tableau / AI 両方の境界拡張の話をしてきましたが、これは決して Tableau Community に閉じた話ではないと思っています。

同じく Tristan Handy が「[BI's Second Unbundling](https://roundup.getdbt.com/p/bis-second-unbundling)」で書いていますが、BI ツールというものが、20年前のフルスタックから、ここ10年で一度アンバンドル（分解）され、今まさに「コーディングエージェントを起点としたフロントエンド」として再構築されつつある、という大きな流れがあります。

Tableau の今回のメッセージ、Tableau Next の方向性、Salesforce Headless 360、それらは別々の発表に見えますが、結局この大きな流れの一断面として読むのが、自分には一番素直な解釈に思えます。

つまり、Tableauユーザーが今直面している変化は、Tableauユーザー固有の問題ではなくて、データに関わる人全体が直面している変化の一形態である、ということです。

---

## 結局、自分たちの「生息地」は Tableau の外に広がっていく

ここまでの話を踏まえて、自分が今、はっきりと感じているのは、「Architect」を本気で受け取れば受け取るほど、自分たちが学ぶべき領域、触るべきツール、接続するべきコミュニティは、Tableau の外に広がっていく、ということです。

たとえば:

- セマンティクス、データモデリング、データエンジニアリング側
- Salesforce / Agentforce / Headless 360 側
- Snowflake、dbt、Anthropic、OpenAI などの周辺エコシステム
- AI Agent、Skills、ハーネス、コンテキストエンジニアリングの語彙

Tableau Community の中だけに生息していると、Tableau の世界観が世界の全てに見えてしまうように思います。でも、本当の「Architect」がやる仕事は、Tableau の外の世界観も含めて設計することではないでしょうか。

そしてこれはたぶん、Tableau社が「やりましょう」と促してくれる種類の話ではないと思います。Tableau社にはTableau社の語り方の制約があって、それは前段の余談で書いた通り、ある程度仕方のないことだと思っています。だからこそ、ユーザー側、つまり自分たちが、自分から境界を引き直していく必要がある、と感じています。

「Tableauユーザー」というアイデンティティはもちろん大事ですが、それに閉じてしまうのは、5年もしないうちに辛くなってくるのではないでしょうか。「データを使って意思決定をデザインする人」「Architectとして動く人」というふうに、自分の中での自己定義をアップデートし続けていく必要があるように思います。

### 余談：「Tableau MCPで何ができるの？」「AIで何ができるの？」と聞かれたら

TC26 の振り返りコメントを SNS や現地で見ていると、「Tableau MCP が何ができるのかもっと知りたい」「AIがどのようなことができるのかもっと知りたい」というような声を散見しました。これに対して、自分が思っていることを2つだけ書かせてください。

**1つ目: 一次情報を読みましょう。読みづらければ、AIに読ませて対話しながら読みましょう**

Tableau MCP の話なら、公式ドキュメントが既に存在します。AI 一般の話も、OpenAI や Anthropic のブログやリリースノートが一次情報です。これらを読みに行くのが、結局一番速いと自分は思っています。

「英語だから」「技術的だから」読みづらい、というのも、今は障害になりません。Claude や ChatGPT に PDF や URL を渡して、「自分はTableauユーザーで、こういうことを知りたい」と話しかければ、対話しながら理解を進められます。むしろ、これをやらないのは勿体ないです。

**2つ目: 「AIで何ができる/できない」という観点は、基本的に捨てて良いと思います**

代わりに、「**どうさせるか**」「**できるようにするために何を設計するか**」という、実現可能性を模索し設計する側の思考に、早く切り替えた方が良いと自分は感じています。

たとえば、TC26 の Keynote でも「89%の組織が AI のデータ利用の不正確性に悩んでいる」みたいな数字が出ていましたが、これは現代では、基本的に設計の問題だと自分は捉えています。モデルの能力の問題というよりは、ハーネス、コンテキスト、Skills、データ整備、そういう周辺設計の問題です。

そして、その設計観点を持つには、今の AI Agent 動向 — OpenAI や Anthropic のブログ、リリース、論文 — を一定見ておく必要があります。これも、結局は前段の話、「生息地を広げる」につながっていきます。

## TC26 全体への感想（個人的な3つの振り返り）

ここまで、「Architect」というキーワードを軸に、構造的な議論をしてきました。最後に、TC26 というイベントそのものを浴びての、もう少し個人的な振り返りを3つだけ書かせてください。

<!-- 画像: TC26 会場やDatafamの集合写真など（任意） -->

**1. ユーザーとの交流は、文句なしに最高だった**

Datafam は常に最高で、元気と刺激を与えてくれます。このネットワークに、自分は値段をつけられないと思っています。これは疑いようがなく、TC に来る一番の理由はここにある、と改めて感じました。

**2. コンテンツや雰囲気については、正直、疑問が残った**

「agentic analytics platform」を目指すと宣言する製品のカンファレンスなのだから、Keynote等の Tableau 主導のコンテンツでも、もっと Agentic の vibes を全面に出してもらえたら、自分はより興奮できたと思います。今は技術的な観点が、意図的に隠されているように見えてしまいました。これは前段の余談で書いた通り、「場の制約」として理解はできるのですが、聴衆の一人としては、率直にもう少し踏み込んでほしいと感じる場面が多かったです。

**3. 自分の中で、二層に分かれる感覚があった**

長年の Tableau ユーザーとしての自分は、TC26 を無限大に楽しめました。そこには確かに大きな興奮があったと思っています。

一方で、Agentic Analytics をどう実現するかを模索し悩んでいる自分としては、もっとコアな話題に触れたい、もっと深いところで議論したい、という飢えのような感覚もありました。

そのコアな議論を求める自分は、もしかしたら Salesforce 系のイベント（TDX、Dreamforce、Agentforce 系のもの）を探した方が良いのかもしれません。あるいは Snowflake、dbt、Claude、それぞれのコミュニティも、もっと模索していきたいと思いました。

これは、まさにこの記事の本論で書いた「生息地を広げる」の体感そのもので、自分自身が TC26 を浴びて、その必要性をリアルに感じてしまった、ということなのだと思います。

## 最後に

この記事を書いた動機は、TC26 の Keynote と一連のセッションを浴びて、頭の中で動き続けている思索を、いったん自分の言葉で言語化しておきたかった、というものでした。

書いてみて改めて感じたのは、Keynote が提示した「Architect」というキーワードは、本気で受け取れば受け取るほど、Tableau の内側だけでは完結しない世界の話につながっていく、ということです。これは「Tableau Classic を捨てよう」という話ではなく、「自分の生息地を、Tableau の外にも広げていく」話だと自分は理解しています。

Tableau Next、Agentforce、dbt、Claude、Headless 360 — 自分は今、これらのほとんどに対して、正直に言って初心者です。ただ、それを「辛い」と感じるよりも、「楽しい」と感じる方を選びたいと思っています。

TC26 で参加した Tristan Handy のセッションは、変化に対して embrace し、心から楽しむ姿勢、というものを、実際に登壇する人の佇まいを通じて見せてくれました。これは現地で参加した一人の人間として、本当に強いインスピレーションをもらえた瞬間でした。自分も、彼のように、これからの変化を心から楽しむマインドセットを持っていたいと思います。

「Architect」になっていく道のりは、たぶん長いし、まだまだ自分にも分からないことだらけです。事例が出てきたら、また考えがまとまったら、この続きをまた書いてみようと思います。

## I. 差分の解釈 (フェーズ1の観察)

### 1. 全体構造 — 章数・粒度

- **AI版**: 大見出し 6章 + サブ見出し 1 + 余談 2 (合計 9 セクション程度)。各章タイトルが論点要約型 (「## ただし「Architect」を本気で受け取ると、Tableau の内側だけでは足りない」)
- **ユーザー版**: 大見出し 3章 + 最後に (合計 4 セクション)。章タイトルは問い・観察型 (「Architectとして動くための、ユーザー側の準備はあるか」「むしろVisionaryが変化の必要性を主張していた」)
- **観察**: AI は論点ごとに章を立てる傾向があり、結果としてセクション数が膨らむ。ユーザー版は近接論点を1章に押し込み、章タイトルを「問い」「観察」の形にして本文展開に余地を残す

### 2. 冒頭の入り方

- **AI版**: 参考リンクをまとめて先出し → プレースホルダー → 「今回書くこと」の宣言段落 (caveat含む) → 本論
- **ユーザー版**: 公開日 → 「TC26 に参加・登壇してきました」と体験から開始 → 登壇したセッション名 (Workout Wednesday Live, AI-Powered Tableau Development and User Experience) → 趣旨1-2文 → caveat 1段落
- **観察**: AI は「整理を見せる」型で冒頭設計するが、ユーザーは「自分の現在地・体験」から入る。参考リンクの先出しはユーザー版にはなく、本文中に散らされている

### 3. 文体・トーン — 語尾と感情語

- **AI版**: 「と感じています」「と思っています」「ように見えます」「気がします」が語尾に均等に並ぶ。共感誘発の語尾はほぼなし
- **ユーザー版**: ヘッジ語尾に加えて「ですよね?」「面白かったです」「気になりました」「興奮しました」を頻繁に挟む。さらに **括弧書きの自己訂正** ("(自分が参加したセッションがそうだった、という話かもしれませんが)"、"(筆者の勘違いである可能性は、もちろん大いにありますが)") を効果的に使う
- **観察**: AI のヘッジは「主張のトーン弱化」のための機械作業。ユーザーのヘッジは「読者を巻き込む」「自分の不確かさを見せる」というコミュニケーションになっている

### 4. 抽象/具体のバランスと順序

- **AI版**: 抽象論 (「Tableau Classic の構造的限界」「両側から境界を広げる」「生息地を広げる」) を先に出し、具体で支える順
- **ユーザー版**: 具体観察 (「MCP使ったことある人?の挙手の少なさ」「Mark の True to the Core の発言」「Will の Keynote が Tableau Next を出さず Claude Code + MCP のデモ主体だった」「Beyond the Boundaries of Tableau の 26:40」) から入り、そこから自然に解釈へ
- **観察**: 順序が真逆。AI は「概念 → 例」、ユーザーは「現地体験 → 解釈」。後者の方が体感的で説得力が強い

### 5. 固有名詞・一次情報の量

- **AI版**: 「Will の Tableau Keynote」「Tristan Handy のセッション」「Adrian の "5 Things..."」と参照はするが、内容説明は薄い
- **ユーザー版**: Mark, Southard, Will という個人名を躊躇せず出す。「Beyond the Boundaries of Tableau (特に26:40から)」とタイムスタンプまで指定。「Tableau MCP でさえリリースから1年も経っていません」のような時間軸の具体性も追加
- **観察**: AI は固有名詞を抽象化して「一般化しやすい引用」に変えがち。ユーザーは個別性を保存し、現地で見た人の証言として扱う

### 6. 製品論 vs ユーザー主体性

- **AI版 (初期)**: 「Tableau Classic はバイナリで Git 不向き」「API-firstじゃない」「Calculated Fields は GUI に埋め込まれてブラックボックス」など、製品の構造的限界を分析
- **AI版 (修正後)**: ユーザー指摘を受けて修正し、「ユーザー側の準備」へ転換。**ただし、ユーザー指摘がないと製品論に戻りがちな性質は残った**
- **ユーザー版**: 一貫して「我々ユーザー側の準備」「Tableau の良さは連携力」というポジティブから入り、「能動的に自身を拡張する姿勢」で締める。製品の欠点列挙はしない
- **観察**: AI は外側 (製品) を分析対象にしがち、ユーザーは内側 (我々) に問題を引き受ける主体性志向

### 7. 余談/補足の使い方

- **AI版**: `### 余談：` 見出しを 2つ使用 (Tableau/Salesforce側への共感的観察 + 「もっと知りたい」へのヒント)
- **ユーザー版**: `### 余談：` 見出しを **1つも使っていない**。同じ内容 ("忖度のような空気" "コンテンツに技術観点が意図的に隠されている") を散文として本論に組み込んでいる
- **観察**: AI は「著者文体らしさのために余談を入れる」発想が強い。ユーザーは本筋に組み込める内容は組み込み、余談セクション化に頼らない

### 8. 結びの設計

- **AI版**: 「## 最後に」を標準型で構成 — 動機振り返り → Tristan インスピレーション → 「初心者として進む」→ サインオフ
- **ユーザー版**: 「## 最後に」を **記事固有の混合型** で構成 — 短い結論宣言 ("能動的に自身を拡張する姿勢") → Knowledge Graph の具体例 → Moving Up the Stack 的自己開示 ("自分もずっと Tableau の世界で幸せに生きられたら良かった") → 5箇条のまとめ (Architect同意 / Datafam / コンテンツへの注文 / 来年への期待 / Agent-native コミットメント) → サインオフ
- **観察**: AI は voice-style.md の「最後に」定型を機械的に踏襲。ユーザーは記事の主張に応じて、複数の型を混合した独自構成にした

### 9. 引用・参考記事の位置

- **AI版**: 冒頭に参考リンク4本をまとめて配置、本文中でも再リンク
- **ユーザー版**: 本文中に分散配置。Anthropic ブログ3本 (Effective context engineering / Harness design / Scaling Managed Agents) は「Architect = 設計重視」を補強する文脈で。dbt 2本 (BI's Second Unbundling / Five Things) は「最近自分はdbtを学んでいて、この方の Blog が好きで」と個人の関心の延長で導入
- **観察**: AI は「情報の構造化」として冒頭にまとめる。ユーザーは個人の関心の流れの中に自然に埋め込む

### 10. メタファー・キーフレーズの使い方

- **AI版**: 「両側から境界を広げる」「生息地を広げる」を記事の柱として何度も参照、章をまたぐ
- **ユーザー版**: 「能動的に自身を拡張する姿勢」「Moving Up the Stack」のような印象的なフレーズを使うが、それぞれ **1回限定** で出して使い捨て。論証の柱にしない
- **観察**: AI はメタファーを論証の柱にしたがる。ユーザーはメタファーを「具体観察の解釈」として1回出すだけで満足

### 11. 章間の遷移マーカー

- **AI版**: 「ここまでは、ほぼ祝福の話です」「ここから少し、踏み込んだ話を」「視野を広げる必要があるのは Tableau の周辺エコシステム側だけではないように感じています」など、丁寧なロードマップ的遷移文を多用
- **ユーザー版**: 「一方で」「ただ」「ということで」と軽い接続詞のみ。章間の道案内文はほぼなし
- **観察**: AI は読者を案内しすぎる。ユーザーは読者を信頼して飛ばす

### 12. 鉤括弧「」の使用頻度

- **AI版**: 「Architect」「Tableau Classic」「Knowledge as Code」「環境整備重視」など頻繁に「」を使う (1段落あたり3-5個出るケースあり)
- **ユーザー版**: 「Architect」「Agentic Analytics Platform」「Help people see and understand data」など本当に強調したい少数のみ (1段落1-2個程度)
- **観察**: AI は「念のため強調しておこう」と過剰に「」を使う。ユーザーは「これは本当に強調すべきか」を自問して使っている様子

## II. AIが改善すべきパターン (フェーズ2の帰納)

1. **整理を見せたがる病** — 章建てを細分化、ロードマップ的遷移文を頻発、参考リンクを冒頭に集約。「読者に構造を見せる」ことを優先しすぎる
2. **抽象メタファーを背負う病** — 一度立てた印象的フレーズを記事の柱として繰り返し参照。具体観察の解釈として使い捨てる発想が薄い
3. **全部を分析モードで書く病** — 観察・感想・興奮を控え、解釈・分析だけで進む。「面白かったです」「興奮しました」のような感情語尾が出ない
4. **製品/対象の構造的限界を書きたがる病** — ユーザー側の主体性を語る方が著者文体に合うのに、つい外側 (製品の欠点) を分析する
5. **ヘッジが機械化する病** — 「と思います」「気がします」を語尾に均等に挿入することで「ヘッジしている気になる」。括弧書き自己訂正や共感語尾との混在が抜ける
6. **読者を案内しすぎる病** — メタな遷移文 (「ここまでは〜、ここからは〜」) が多くなり、文章密度が薄まる
7. **固有名詞を抽象化する病** — 「Will の Keynote」と一般化してしまう。タイムスタンプや個人名による現地感が消える
8. **余談セクション化に頼る病** — 散文に組み込める内容まで `### 余談：` で囲ってしまう。「著者文体らしさ」と誤解している
9. **「最後に」標準型踏襲病** — 動機 → 再認識 → 開示 → 続編 → サインオフ、の定型を機械的に踏襲する。記事固有の畳み方 (5箇条まとめ等) を選ばない
10. **鉤括弧過剰使用病** — 「念のため」「保険として」通常用語まで「」で囲う

## IV. メタコメント

### 1. voice-style.md の二層構造化は機能している

ルール 1-16 を「従うべき型」、17-21 を「AI 的傾向への対抗ガイド」と分離した構造は、今回の比較を見る限り **必要な観察を全てカバーできている**。

ただし、本当に効くかは「voice-style.md を読んだ後に AI が書く draft が、ユーザー版にどれだけ近付くか」の **次回検証** が必要。今回はあくまで「先に更新されたルールを後から検証している」だけで、ルール → 生成の流れは未検証。

### 2. このスキルの位置付け

`/compare-drafts` スキルは、**voice-style.md を継続的に育てるためのフィードバックループの一部** として機能する。

- 1記事ごとに `comparison.md` を生成
- 複数記事の `comparison.md` を横断的に見れば、繰り返し現れるパターンは確実に「voice-style.md に必要なルール」と判定できる
- 1記事限りの特殊性 (記事固有の話) はルール化せず、観察として残すだけで良い

### 3. 観察への注意

今回の比較で抽出されたパターンは「TC26 思索」という **感想記事** に強く依存している可能性がある。技術解説記事や手順記事だと、別のパターン (例: 抽象論先行が許される、章建てが多くても OK、参考リンク冒頭まとめが有効) が出るかもしれない。

ルール 17-21 が「すべての記事タイプで成立する」と早合点せず、**異なるタイプの記事で比較を蓄積してから一般化** する慎重さが望ましい。

### 4. 次の検証チャンス

- voice-style.md 更新後、別記事を `/write` で生成 → `/compare-drafts` で比較 → ルール 17-21 が本当に効いたかを確認
